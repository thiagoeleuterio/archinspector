# Rule ID

HEX-010

# Title

Messaging concerns must remain behind adapters

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether brokers, queues, and message handlers interact with the application core through ports rather than leaking messaging concerns into core behavior within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes application core behavior and one or more messaging mechanisms, message handlers, publishers, subscribers, brokers, queues, topics, or messaging adapters. The rule is relevant only when messaging interactions can be distinguished from core behavior and port-based boundaries.

# Rule Statement

Messaging concerns must remain behind adapters.

# Rationale

Messaging concerns in the application core couple core behavior to broker contracts, delivery mechanics, message transport assumptions, and handler infrastructure. Keeping messaging behind adapters allows the core to express application behavior through ports while adapters translate between messaging mechanisms and core interactions.

# Evidence Required

Evaluation requires traceable evidence that identifies the application core scope, the reviewed messaging concerns, the adapters or message-facing components, and the ports or equivalent boundaries used to interact with the core. Direct evidence may include project or module references, package references, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, message handler declarations, broker or queue configuration, or reviewed code excerpts showing whether messaging mechanisms interact with the core through ports or leak into core behavior. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between application core behavior, ports, messaging adapters, and messaging mechanisms. Then evaluate whether message handlers and broker-facing components translate messages and delegate through ports, and whether core behavior remains independent from broker APIs, queue semantics, transport metadata, message handler infrastructure, and messaging configuration. Interpret evidence conservatively, and do not infer messaging leakage from naming, intended architecture, or the mere presence of asynchronous behavior. This rule evaluates messaging as an adapter boundary concern, not event semantics or message design quality.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant messaging interactions and the available evidence supports that brokers, queues, message handlers, and messaging infrastructure interact with the application core through ports while messaging concerns remain behind adapters.

# Fail Conditions

Use `Fail` when direct evidence shows that application core behavior depends on broker APIs, queue mechanisms, transport metadata, message handler infrastructure, messaging configuration, concrete messaging adapters, or messaging-specific contracts instead of interacting through ports.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, or ambiguous messaging concern leakage into the application core, or inconsistent use of ports by messaging adapters, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include application core behavior, messaging mechanisms, message handlers, messaging adapters, brokers, queues, topics, or messaging interactions relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the application core scope, identify messaging concerns, identify ports or equivalent boundaries, inspect how messaging components interact with the core, determine whether messaging concerns remain behind adapters, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when messaging concerns shape central application behavior, broad message-driven flows, core contracts, delivery assumptions, or stable architectural boundaries. Assign lower severity when the leakage is narrow, isolated, indirect, or limited to peripheral messaging interactions. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant core scope, messaging concern, adapter boundary, port interaction, and dependency relationship. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.
