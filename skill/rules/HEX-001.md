# Rule ID

HEX-001

# Title

Domain layer must not depend on infrastructure

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether domain code remains independent from infrastructure concerns within the reviewed scope.

# Applicability

Apply this rule when the reviewed material provides enough context to identify a domain scope and a distinct infrastructure scope or infrastructure concern. The rule may be considered regardless of whether the system explicitly claims a named architecture style.

# Rule Statement

The domain layer must not depend on infrastructure.

# Rationale

Infrastructure dependencies in domain code make domain behavior harder to reason about independently, increase coupling to technical mechanisms, and reduce the ability to change external integration concerns without affecting domain logic.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed domain scope and the reviewed infrastructure scope or infrastructure concern. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, or reviewed code excerpts. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between domain and infrastructure. Then evaluate the direction of dependencies using the strongest available evidence. Interpret evidence conservatively, and do not infer dependency direction from intended architecture, naming alone, or missing material. A dependency should be considered relevant only when it connects domain code to infrastructure code, infrastructure abstractions, or infrastructure concerns within the reviewed scope.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant dependencies and the available evidence supports that domain code does not depend on infrastructure.

# Fail Conditions

Use `Fail` when direct evidence shows that domain code depends on infrastructure code, infrastructure abstractions, or infrastructure concerns within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates a partial, indirect, inconsistent, or ambiguous dependency from domain code toward infrastructure, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include a domain layer, domain scope, or distinct infrastructure concern relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the domain scope, identify the infrastructure scope, inspect dependency direction, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when the dependency affects central domain behavior, broad domain scope, or stable architectural boundaries. Assign lower severity when the dependency is narrow, isolated, or limited to peripheral domain code. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant scopes and dependency direction. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.
