# Rule ID

HEX-004

# Title

Application core must use outbound ports for external systems

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether application core behavior depends on outbound ports rather than concrete external system integrations within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes application core behavior that needs to interact with external systems, infrastructure mechanisms, or technical integrations. The rule is relevant only when outbound interactions can be identified within the reviewed scope.

# Rule Statement

The application core must use outbound ports for external systems.

# Rationale

Outbound ports protect core behavior from concrete external integrations, preserve dependency direction toward the core, and allow external systems to change without forcing corresponding changes in application or domain behavior.

# Evidence Required

Evaluation requires traceable evidence that identifies the application core scope and the reviewed outbound interactions with external systems. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, or reviewed code excerpts showing whether core behavior depends on outbound abstractions or concrete integration details. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between application core behavior and external system interactions. Then evaluate whether the core expresses outbound needs through stable ports or depends directly on concrete integrations, external APIs, infrastructure mechanisms, or technical details. Interpret evidence conservatively, and do not infer outbound port usage from intended architecture, naming alone, or missing material.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant outbound interactions and the available evidence supports that application core behavior uses outbound ports for external systems.

# Fail Conditions

Use `Fail` when direct evidence shows that application core behavior depends directly on concrete external system integrations, infrastructure mechanisms, external APIs, or technical details instead of outbound ports.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, implicit, or ambiguous outbound port usage, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include application core behavior or outbound interactions with external systems relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the application core scope, identify outbound interactions, inspect dependency direction, determine whether outbound ports exist, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when direct core dependency on external systems affects central application behavior, broad outbound integration surfaces, or stable architectural boundaries. Assign lower severity when the dependency is narrow, isolated, or limited to peripheral integration paths. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant scopes, outbound interactions, and dependency direction. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.
