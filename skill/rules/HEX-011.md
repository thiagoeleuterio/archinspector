# Rule ID

HEX-011

# Title

Adapter models must not leak into core contracts

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether transport, persistence, or infrastructure DTOs remain outside core boundary contracts within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes application core contracts, inbound or outbound ports, equivalent boundary abstractions, and one or more adapter-facing models used by transport, persistence, or infrastructure concerns. The rule is relevant only when there is enough context to distinguish core boundary types from adapter models.

# Rule Statement

Adapter models must not leak into core contracts.

# Rationale

When adapter models become part of core contracts, the application core boundary is shaped by transport, persistence, or infrastructure concerns. This couples core behavior to external mechanisms, weakens adapter substitutability, and makes technical model changes more likely to affect application-facing contracts.

# Evidence Required

Evaluation requires traceable evidence that identifies the application core scope, the reviewed core contracts or equivalent boundary abstractions, and the transport, persistence, or infrastructure models related to adapters. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method signatures, method behavior, or reviewed code excerpts showing whether core contract members use adapter model types. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between core contracts and adapter models. Then evaluate whether types used by core contracts express core boundary needs or instead come from transport, persistence, or infrastructure concerns. Interpret evidence conservatively, and do not classify a type as an adapter model based only on naming or location. This rule evaluates adapter model leakage into core contracts; ownership of ports, general dependency direction, persistence mechanisms, messaging mechanisms, and framework leakage should be evaluated by their corresponding rules when applicable.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant core contracts and adapter models, and the available evidence supports that transport, persistence, or infrastructure DTOs do not leak into core boundary contracts.

# Fail Conditions

Use `Fail` when direct evidence shows that core contracts, ports, or equivalent boundary abstractions expose transport DTOs, persistence DTOs, infrastructure DTOs, or other adapter models as contract input types, output types, member types, or required boundary types within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, or ambiguous adapter model leakage into core contracts, or when a core contract type appears partly shaped by adapter models, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include application core contracts, ports, equivalent boundary abstractions, adapter models, transport concerns, persistence concerns, or infrastructure concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the application core scope, identify core contracts, identify adapter models, inspect contract member types, distinguish adapter models from core boundary types, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when adapter model leakage affects central core contracts, broad inbound or outbound surfaces, multiple adapters, or stable architectural boundaries. Assign lower severity when the leakage is narrow, isolated, indirect, or limited to peripheral boundary contracts. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant core scope, core contracts, adapter model identity, and contract type usage. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

HEX-002, HEX-004, HEX-006, HEX-007, HEX-009, HEX-010. These are conceptual dependencies for shared inbound, outbound, port, adapter, persistence, messaging, and dependency-direction vocabulary, not procedural prerequisites for evaluating this rule.
