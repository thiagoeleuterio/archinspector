# Rule ID

HEX-009

# Title

Persistence concerns must remain behind outbound ports

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether storage mechanisms remain behind outbound ports and do not shape domain or application behavior directly within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes domain or application core behavior that interacts with persistent storage, data access mechanisms, persistence adapters, or storage-facing abstractions. The rule is relevant only when persistence interactions can be distinguished from core behavior and outbound boundaries.

# Rule Statement

Persistence concerns must remain behind outbound ports.

# Rationale

Allowing storage mechanisms to shape domain or application behavior couples core decisions to persistence details, makes storage changes more disruptive, and can cause the core boundary to reflect database structures or data access APIs instead of application needs.

# Evidence Required

Evaluation requires traceable evidence that identifies the domain or application core scope, the reviewed persistence concerns, and the outbound ports or equivalent boundary abstractions used for storage interactions. Direct evidence may include project or module references, package references, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, persistence configuration, data access contracts, or reviewed code excerpts showing whether storage mechanisms remain behind outbound ports or directly shape core behavior. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between core behavior, outbound ports, persistence adapters, and storage mechanisms. Then evaluate whether core behavior expresses persistence needs through outbound ports rather than depending on storage APIs, persistence models, query mechanisms, schema details, or data access configuration. Interpret evidence conservatively, and do not infer persistence leakage from naming, intended architecture, or the mere existence of stored data concepts. This rule evaluates persistence as a hexagonal boundary concern, not domain modeling quality or general data design.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant persistence interactions and the available evidence supports that persistence concerns remain behind outbound ports and do not directly shape domain or application behavior.

# Fail Conditions

Use `Fail` when direct evidence shows that domain or application core behavior depends on storage APIs, persistence models, query mechanisms, schema details, data access configuration, or concrete persistence adapters instead of outbound ports.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, or ambiguous persistence concern leakage into domain or application behavior, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include domain or application core behavior, persistent storage interactions, persistence concerns, outbound ports, or storage-facing abstractions relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the domain or application core scope, identify persistence concerns, identify outbound ports or equivalent abstractions, inspect how storage interactions are performed, determine whether persistence shapes core behavior, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when persistence concerns shape central domain or application behavior, broad storage interaction surfaces, core contracts, or stable architectural boundaries. Assign lower severity when the leakage is narrow, isolated, indirect, or limited to peripheral persistence interactions. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant core scope, persistence concern, outbound boundary, and dependency or shaping relationship. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

HEX-004, HEX-005, HEX-006, HEX-007. These are conceptual dependencies for shared outbound port, adapter, ownership, and dependency-direction vocabulary, not procedural prerequisites for evaluating this rule.
