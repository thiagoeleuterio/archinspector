# Rule ID

HEX-003

# Title

Inbound adapters must not contain core business behavior

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether inbound adapters remain delivery-facing components that delegate core business behavior to the application core within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes inbound adapters, delivery mechanisms, or external-facing components that receive input and invoke application or domain behavior. The rule is relevant only when there is enough context to distinguish inbound adapter responsibilities from application core responsibilities.

# Rule Statement

Inbound adapters must not contain core business behavior.

# Rationale

Placing core business behavior inside inbound adapters couples application decisions to delivery mechanisms, makes behavior harder to reuse across entry points, and weakens the boundary that protects the application core from external interaction concerns.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed inbound adapters or delivery mechanisms and the application core behavior relevant to them. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, or reviewed code excerpts showing whether inbound adapters translate and delegate input or perform core business behavior themselves. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between inbound adapter responsibilities and application core responsibilities. Then evaluate whether behavior located in inbound adapters is limited to delivery concerns or whether it represents core business decisions, orchestration, or rules that should belong to the application core. Interpret evidence conservatively, and do not classify behavior as core business behavior based only on naming, file location, or assumed intent.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant inbound adapters and the available evidence supports that they delegate core business behavior to the application core while retaining only inbound delivery responsibilities.

# Fail Conditions

Use `Fail` when direct evidence shows that inbound adapters contain core business behavior, central application decisions, or domain-relevant rules instead of delegating them to the application core.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, duplicated, or ambiguous business behavior in inbound adapters, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include inbound adapters, delivery mechanisms, external-facing components, or core business behavior relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify inbound adapters, distinguish adapter responsibilities from core responsibilities, inspect behavior placement, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when core business behavior in inbound adapters affects central application flows, multiple entry points, or stable architectural boundaries. Assign lower severity when the behavior is narrow, isolated, duplicated at a small boundary, or limited to peripheral flows. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant scopes, adapter responsibilities, and behavior placement. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.
