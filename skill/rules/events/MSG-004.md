# Rule ID

MSG-004

# Title

Event granularity

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether event or message granularity is coherent with the represented fact, consumer needs, and acceptable coupling.

# Applicability

Apply this rule when reviewed material includes events or messages emitted or consumed as architectural communication units and there is enough scope to inspect their payload, emission point, consumer interpretation, or documented responsibility.

Do not apply this rule merely because a message is small, large, coarse, fine, or named generically. The rule is not applicable when the reviewed material contains no event or message communication unit whose granularity affects architecture.

# Rule Statement

An event or message must represent a coherent communication responsibility at a granularity that does not unnecessarily merge independent facts, fragment one fact into noisy messages, or force consumers to depend on irrelevant producer detail.

# Rationale

Granularity affects coupling and interpretability. Overly broad messages can make consumers parse irrelevant changes or depend on full producer state. Overly fine messages can create event-per-setter noise, ordering pressure, excessive choreography, or fragmented consumer reasoning. The correct granularity depends on context and evidence, not a universal size.

# Evidence Required

Evaluation requires evidence of message payload, emission conditions, represented fact or responsibility, consumer behavior, handler logic, projections, integration contracts, tests, and documentation where available.

Primary evidence includes producer emission points, payload schemas, state changes that trigger messages, consumers and projections, filtering logic, business process reactions, tests, and ADRs. Message names and field counts are supporting evidence only.

# Evaluation Guidance

Evaluate whether the message has one coherent reason to be published or consumed. Look for independent facts merged into one message, broad entity snapshots used where consumers need a narrower fact, trivial messages emitted for every setter, or excessive fragmentation that shifts reconstruction burden to consumers.

Common finding shapes include generic change events requiring consumers to infer meaning, mega-events carrying unrelated state, event-per-property streams with no architectural need, one message mixing independent business outcomes, and fragmented event sets that require consumers to assemble a basic fact from multiple messages.

Remediation may include splitting independent facts, narrowing payload to consumer-relevant contract data, combining overly fragmented messages into a coherent fact, documenting intended granularity, or aligning emission points with meaningful changes. Do not prescribe a universal payload size, event count, or one-event-per-operation rule.

Boundaries: `MSG-001` evaluates domain fact semantics; `MSG-002` evaluates integration boundary; `MSG-006` evaluates naming and meaning; compatibility, ordering, replay, and schema evolution are separate.

# Pass Conditions

Use `Pass` when evidence shows that the message granularity is coherent with the represented fact or integration responsibility and does not impose unnecessary interpretation, coupling, or fragmentation on consumers.

# Fail Conditions

Use `Fail` when direct evidence shows that granularity creates real architectural harm through merged independent facts, excessively broad payloads, event-per-trivial-change noise, or fragmentation that consumers must compensate for.

# Warning Conditions

Use `Warning` when granularity appears inconsistent, overly generic, overly broad, or overly fine, but evidence of architectural harm or consumer impact is partial or ambiguous.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope has no event or message communication units, or when message granularity cannot affect the reviewed architectural concern.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may apply but the reviewed material does not show emission points, payload content, represented facts, consumer needs, or behavior sufficient to evaluate granularity.

# Severity Guidance

Severity depends on affected consumer count, coupling, ordering pressure, processing cost, criticality of misinterpreted facts, recurrence, retained-message impact, and ease of correction. Do not assign severity from payload size alone.

# Confidence Guidance

Use `Confirmed` when direct evidence shows payload, emission behavior, consumer usage, and granularity impact. Use `Likely` when multiple signals support the conclusion with incomplete consumer or producer evidence. Use `Possible` for limited or indirect signals. Use `Not Enough Evidence` when granularity impact cannot be traced. Naming alone must not produce `Confirmed`.
