# Rule ID

MSG-018

# Title

Poison message handling

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether permanently non-processable messages are identified, isolated, and prevented from repeatedly degrading or blocking normal processing.

# Applicability

Apply this rule when consumers may receive invalid payloads, unsupported contract versions, corrupt data, impossible state, deterministic validation failures, or messages that cannot succeed through ordinary retry.

Do not apply this rule when message failures are only transient in the reviewed scope or when no consumer path can encounter permanently non-processable messages.

# Rule Statement

Permanently non-processable messages must be distinguishable from transient failures and isolated or resolved so they do not repeatedly block partitions, exhaust retries, consume resources, or degrade normal message flow.

# Rationale

Retrying a message that can never succeed can create repeated failures, backlog growth, partition blocking, noisy alerts, and service degradation. Poison handling protects normal flow by classifying, isolating, diagnosing, correcting, or discarding such messages intentionally.

# Evidence Required

Evaluation requires evidence of validation behavior, error classification, retry decision logic, poison detection, isolation or quarantine paths, dead-letter interaction, diagnostics, discard or repair decisions, reprocessing procedures, tests, and operational records.

Primary evidence includes consumer code, schema validation, exception handling, retry classification, broker settings, failed-message records, logs, metrics, runbooks, and tests.

# Evaluation Guidance

Evaluate whether deterministic or permanent failure is recognized. A failed message is not automatically poison; the rule concerns messages that cannot succeed without data correction, contract support, code change, or controlled discard.

Common finding shapes include retrying malformed payloads forever, blocking a partition on one invalid message, sending poison messages back to the same loop, no distinction between transient and permanent failure, missing diagnostics for invalid payloads, and uncontrolled discard of corrupt messages.

Remediation may include classifying permanent failures, validating before side effects, quarantining poison messages, preserving diagnostics, defining correction or discard criteria, preventing partition starvation, and coordinating with dead-letter handling. Do not require a specific quarantine technology.

Boundaries: `MSG-008` covers compatibility breaks that can create permanent processing failures. `MSG-009` covers schema evolution drift and invalid payload structure. `MSG-016` covers retry policy; `MSG-017` covers dead-letter destination and operational resolution; this rule covers poison identification and isolation.

# Pass Conditions

Use `Pass` when evidence shows permanently non-processable messages are identified, isolated or resolved, and prevented from continuously blocking or degrading normal processing.

# Fail Conditions

Use `Fail` when direct evidence shows poison messages can retry indefinitely, block processing, loop through failure paths, lose diagnostic context, or be discarded unsafely with real impact.

# Warning Conditions

Use `Warning` when poison handling is partial, generic, weakly classified, dependent on manual inspection, or not fully evidenced for important failure modes.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope has no realistic permanent message failure path or only transient failures are relevant.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when material does not show validation, error classification, retry behavior, isolation, diagnostics, discard, correction, or operational handling.

# Severity Guidance

Severity depends on ability to block partitions or consumers, backlog impact, resource exhaustion, business criticality, data corruption risk, recovery cost, observability, and affected flows.

# Confidence Guidance

Use `Confirmed` when direct evidence shows permanent failure behavior and isolation or lack of it. Use `Likely` when multiple signals align with incomplete operational evidence. Use `Possible` for limited signals. Use `Not Enough Evidence` when failure classification cannot be inspected. Generic exception handling alone must not produce `Confirmed`.

# Dependencies

MSG-008, MSG-009, MSG-016, MSG-017. These are adjacent responsibilities, not procedural prerequisites.
