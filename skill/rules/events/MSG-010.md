# Rule ID

MSG-010

# Title

Producer-persistence consistency

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether durable state changes and the decision to produce a related event or message remain consistent across success, failure, rollback, and recovery paths.

# Applicability

Apply this rule when a produced message represents, announces, or depends on a durable state change, persisted business decision, or stored fact.

Do not apply this rule to messages unrelated to persisted state, fully volatile flows where loss is acceptable and documented, or scopes that only inspect consumer delivery after the producer handoff.

# Rule Statement

When a message depends on a durable state change, the architecture must prevent or recover from inconsistent outcomes where state is persisted without publication intent or a message is published without the corresponding durable state.

# Rationale

Producer-side dual writes can leave consumers with missing facts or false facts. A failure between database commit and publish, publish and commit, rollback and send, or recovery retry can create observable inconsistency, duplicate publication, or lost publication intent.

This rule evaluates the consistency problem and acceptable strategies. It does not require Transactional Outbox universally.

# Evidence Required

Evaluation requires evidence of persistence operations, publication decision, transaction boundaries, rollback behavior, failure handling, retry paths, recovery behavior, producer code, tests, and operational records where available.

Primary evidence includes executable producer flow, database or durable write boundaries, message send calls, transaction handling, durable intent records, change capture, retry behavior, failure tests, logs, and ADRs.

# Evaluation Guidance

Inspect whether the state change and publication intent can diverge under realistic failure. Equivalent solutions may include atomic transaction capture, change data capture, idempotent recovery, broker transaction support, compensating process, or documented acceptance of loss when impact is acceptable.

Common finding shapes include state committed before send with no recoverable intent, message sent before state commit, rollback after send, retry producing duplicate observable facts without controls, lost publication after process crash, and tests covering only the happy path.

Remediation may include capturing publication intent durably, aligning commit and send ordering with recovery, making recovery idempotent, documenting accepted loss, or using an existing transactional mechanism. Do not require a named outbox or specific storage.

Boundaries: `MSG-011` evaluates the durable publication record when adopted; `MSG-012` evaluates delivery after handoff; consumer idempotency and event semantics are separate.

# Pass Conditions

Use `Pass` when evidence shows state persistence and publication intent remain consistent or recoverable under relevant failure paths for the reviewed context.

# Fail Conditions

Use `Fail` when direct evidence shows a real unrecoverable dual-write window, rollback/send inconsistency, lost publication intent, or false publication risk tied to durable state.

# Warning Conditions

Use `Warning` when consistency is partially addressed, depends on undocumented assumptions, lacks failure-path evidence, or leaves limited divergence risk without enough evidence for `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when messages do not depend on durable state, loss is legitimate for the flow, or the reviewed scope concerns only delivery or consumption after publication.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when material does not show persistence boundaries, publication timing, rollback behavior, recovery, or failure paths needed to evaluate consistency.

# Severity Guidance

Severity depends on state criticality, loss or false-publication impact, affected consumers, duplicate risk, recovery cost, financial or regulatory exposure, detectability, and operational repair capability.

# Confidence Guidance

Use `Confirmed` when direct evidence shows the persistence/publication flow and failure behavior or gap. Use `Likely` when multiple signals support the conclusion but recovery evidence is incomplete. Use `Possible` for limited flow evidence. Use `Not Enough Evidence` when failure windows cannot be evaluated. Pattern naming alone must not produce `Confirmed`.
