# Rule ID

MSG-009

# Title

Schema evolution

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether structural and semantic changes in message schemas are planned, interpretable, and operable over time.

# Applicability

Apply this rule when events or messages have explicit or implicit schemas that change, may change, are retained, are consumed by more than one code path, or are documented as contracts.

Do not apply this rule when the reviewed scope contains no message schema, no evolution concern, or only local transient payloads whose structure is changed atomically with all usage.

# Rule Statement

Message schema evolution must preserve interpretable structural and semantic change history so producers, consumers, transformations, and operators can understand how fields, types, requiredness, cardinality, enums, units, and meanings change over time.

# Rationale

Schemas can evolve safely only when changes are understood. A field can keep its name while changing meaning, units, requiredness, or cardinality; a payload can change shape without an explicit schema file. Unmanaged evolution makes retained messages, independent consumers, transformations, and diagnostics unreliable.

# Evidence Required

Evaluation requires evidence of schema definitions or implicit payload structures, change history, serializers, transformations, migrations, documentation, tests, consumer behavior, retained data, and producer behavior.

Primary evidence includes schemas, contracts, message classes, serialization configuration, migration or transformation code, compatibility documents, tests over historical payloads, and documentation tied to implementation. A registry is not required.

# Evaluation Guidance

Evaluate the nature and management of schema change, not the mere existence of versions or compatibility outcomes. Look for additions, removals, renames, type changes, requiredness changes, cardinality changes, enum changes, unit changes, semantic changes, transformations, and documentation drift.

Common finding shapes include implicit schemas changing without traceability, documentation diverging from actual payloads, field meaning changes without migration, enum expansion that consumers cannot interpret, required fields added without defaults, and historical payloads that no longer deserialize or mean what consumers assume.

Remediation may include documenting schema history, making implicit schemas explicit enough for review, adding transformation paths, preserving historical interpretation, aligning documentation with code, or testing historical payloads. Do not require a specific registry, format, broker, or schema language.

Boundaries: `MSG-007` owns version identity, `MSG-008` owns compatibility relationship, and this rule owns structural and semantic change mechanics.

# Pass Conditions

Use `Pass` when evidence shows schema changes are understandable, traceable, and operable for the reviewed context, including relevant structural and semantic changes and any needed transformations or migration paths.

# Fail Conditions

Use `Fail` when direct evidence shows unmanaged schema evolution causing real risk, such as incompatible structural or semantic changes, undocumented drift, or historical payloads that cannot be interpreted when they must be.

# Warning Conditions

Use `Warning` when schema evolution is partially documented, implicit, inconsistent, or weakly tested, but evidence does not prove a definitive architectural break.

# Not Applicable Conditions

Use `Not Applicable` when no message schema or schema evolution concern exists in the reviewed scope.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when schema shape, history, serializers, consumers, transformations, retained messages, or documentation are insufficient to evaluate evolution.

# Severity Guidance

Severity depends on retained-message lifetime, consumer count, semantic break risk, criticality, migration difficulty, data loss, detectability, and operational recovery. Do not assign severity from schema format or absence of a registry.

# Confidence Guidance

Use `Confirmed` when direct evidence shows schema changes and their management or failure. Use `Likely` when several sources align with some history or consumer gaps. Use `Possible` for limited or indirect schema signals. Use `Not Enough Evidence` when evolution cannot be reconstructed. Naming alone must not produce `Confirmed`.
