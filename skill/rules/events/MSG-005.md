# Rule ID

MSG-005

# Title

Event immutability

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether published or retained events preserve their content and meaning against improper mutation after they represent a fact.

# Applicability

Apply this rule when reviewed material includes events or messages that are published, persisted, retained, replayed, audited, or consumed asynchronously as records of what was communicated.

Do not apply this rule to transient commands, local request objects, mutable work-in-progress builders before publication, logs not treated as event facts, or message-like structures whose mutation cannot affect a published or retained fact.

# Rule Statement

Once an event or message is published or made available as a fact, its content and meaning must not be mutable by producers, consumers, shared references, later enrichment, or operational updates in a way that changes what was communicated.

# Rationale

Consumers, audits, retries, replay, and diagnostics depend on a stable message record. If the same event instance, payload object, stored record, or external reference can change after publication, downstream reasoning may depend on a fact that no longer matches what was originally communicated.

This rule does not require a specific language feature such as records, readonly properties, freeze operations, immutable collections, or a particular storage model.

# Evidence Required

Evaluation requires evidence of event construction, publication point, payload mutability, object references, storage records, consumer behavior, update paths, enrichment flows, repair tools, audit history, and replay or reprocessing assumptions where relevant.

Primary evidence includes executable code, serialization behavior, persistence updates, consumer mutation paths, tests, schema contracts, operational repair procedures, and documentation. Naming or type style is supporting evidence only.

# Evaluation Guidance

Distinguish pre-publication construction from post-publication mutation. A builder or mutable assembly step may be legitimate before the event becomes available. The concern is mutation that changes content or meaning after publication, retention, acknowledgement, or consumer access.

Common finding shapes include mutable properties modified after publish, consumers mutating shared event instances, payloads holding references to mutable domain objects, delayed filling of required facts, stored event records overwritten without preserving history, and replay reading current external state instead of the recorded fact.

Remediation may include copying payload data before publication, separating builders from published messages, preventing consumer mutation, storing immutable snapshots of communicated facts, preserving corrections as new records, or documenting controlled repair semantics. Do not require one implementation mechanism.

Boundaries: `MSG-001` evaluates whether a Domain Event is semantically completed; this rule evaluates technical and contractual preservation. `MSG-009` evaluates schema evolution; `MSG-020` evaluates replay safety.

# Pass Conditions

Use `Pass` when evidence shows published or retained events preserve content and meaning after publication and cannot be improperly changed through shared references, consumer mutation, persistence updates, or later enrichment.

# Fail Conditions

Use `Fail` when direct evidence shows post-publication mutation, overwriting, shared mutable references, delayed required data, or consumer modification that changes the communicated fact with architectural impact.

# Warning Conditions

Use `Warning` when event immutability is partially protected, inconsistently enforced, dependent on convention, or unclear in some flows, but direct evidence does not justify `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope contains no published, retained, replayed, audited, or asynchronously consumed message facts whose mutation matters.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may apply but material does not show publication timing, object lifecycle, storage behavior, payload copying, consumer access, or mutation paths.

# Severity Guidance

Severity increases when mutation affects audit, replay, critical decisions, many consumers, retained streams, regulatory records, financial flows, or recovery. Severity is lower when mutation risk is localized before publication or affects non-critical internal messages.

# Confidence Guidance

Use `Confirmed` when direct evidence shows lifecycle, publication point, and mutation protection or violation. Use `Likely` when several sources support the conclusion with gaps. Use `Possible` for limited or indirect signals. Use `Not Enough Evidence` when mutation paths cannot be inspected. Type naming alone must not produce `Confirmed`.
