# Rule ID

MSG-001

# Title

Domain event semantics

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether a domain event represents a completed, domain-significant business fact expressed in language that is meaningful within the reviewed domain.

# Applicability

Apply this rule when the reviewed material includes a class, record, struct, schema, contract, message, payload, object, document, record, or other structure that is presented, named, documented, produced, raised, stored, or consumed as a Domain Event.

The rule may also apply when a component produces internal event-like objects with domain-semantic intent, even if the implementation does not use a formal Domain-Driven Design style, a marker interface, an event bus, a broker, a framework, a base class, an annotation, or a specific suffix.

Do not apply this rule merely because the system contains messages, notifications, callbacks, audit records, logs, persistence hooks, integration messages, transport envelopes, commands, DTOs, or technical events. The rule is relevant only when the reviewed evidence supports that the element is intended to represent a domain event or a domain-significant occurrence.

The absence of Domain Events is not a violation by itself. Use `Not Applicable` when the system does not use the concept, when the reviewed scope contains no domain events, when the element is explicitly technical, when the element is a command, when the element is an integration message without Domain Event intent, or when absence of Domain Events is intentional and valid for the reviewed context.

# Rule Statement

A Domain Event must represent a completed, stable, domain-significant fact that has occurred in the domain, using domain language and content sufficient to express that fact without depending on commands, technical operations, generic payload wrappers, mutable external state, or messaging transport mechanisms for its meaning.

# Rationale

Domain events allow a system to make important domain occurrences visible after they happen. When an event represents an intention, command, technical notification, persistence operation, generic data wrapper, or transport concern instead of a completed domain fact, consumers may react to information that is false, premature, ambiguous, or unrelated to business meaning.

The architectural risk is semantic rather than mechanical. Incorrect domain-event semantics can blur the distinction between decisions and facts, couple consumers to producer implementation details, make business processes react to attempted rather than completed changes, and make reviewed evidence look more domain-oriented than the actual behavior supports.

This rule does not require a system to use Domain Events, Domain-Driven Design, asynchronous messaging, event sourcing, a broker, an outbox, an aggregate root as the universal event source, or any particular framework. It evaluates only whether something treated as a Domain Event carries valid domain-event meaning.

# Evidence Required

Evaluation requires traceable evidence identifying the event-like element, its claimed or inferred Domain Event role, the behavior that creates or raises it, the domain decision or state transition associated with it, and the fact it is meant to represent.

Primary evidence includes code that creates, records, raises, publishes, or stores the event; behavior immediately before creation; validation and decision logic preceding creation; associated state transitions; tests demonstrating when the event is produced; internal consumers reacting to the event; contracts or schemas that define event content; documentation or ADRs that describe event meaning; and reviewed domain terminology that explains the event's business significance.

Evidence should establish whether the event describes something that has already happened, whether the fact is relevant to a business rule, business process, domain decision, internal reaction, integration need, semantic audit, projection, meaningful notification, or consistency concern, and whether the payload is coherent with that fact.

Evidence should distinguish Domain Events from commands, requests, technical events, persistence notifications, cache invalidation signals, transport messages, job signals, framework callbacks, exception records, generic message envelopes, DTOs, and integration contracts. Names, suffixes, folders, marker interfaces, comments, diagrams, and routing conventions are supporting signals only and must not produce `Confirmed` confidence without behavioral, structural, contractual, test, or documentation evidence tied to the event's role.

When evidence exists at multiple levels, prefer the following order of strength:

1. Event creation or raising code and the behavior immediately preceding it.
2. Domain decision, validation, or state transition associated with creation.
3. Behavioral tests proving the event is created only when the represented fact has occurred.
4. Consumer behavior that reacts to the fact as already true.
5. Contracts, schemas, and payload definitions.
6. Architectural documentation and ADRs tied to the implementation.
7. Names, folders, suffixes, marker interfaces, comments, and diagrams as auxiliary support only.

# Evaluation Guidance

First determine whether the reviewed element is being treated as a Domain Event. If the element is explicitly a command, technical event, integration message, transport envelope, audit record, log entry, DTO, or internal notification with no Domain Event intent, the rule is usually `Not Applicable` rather than failed.

Evaluate the semantics of the event, not the full domain model. The question is whether the event represents a completed, stable, domain-significant fact in language the domain can understand. The rule may use entities, aggregates, domain services, application flows, tests, and consumers as evidence, but the conclusion must remain about the event's meaning.

Distinguish fact from intent. A Domain Event describes something that happened; a command requests that something happen; a technical request asks infrastructure or integration code to perform an operation. Adding an event suffix or placing an object in an event folder does not transform a command, request, or technical operation into a Domain Event.

Inspect the name, structure, creation moment, preceding behavior, following behavior, producer flow, and consumer interpretation together. A name that sounds event-like is not enough. A name that is grammatically imperfect should not produce a finding when the reviewed flow and payload clearly express a completed domain fact.

Evaluate domain language without enforcing one human language, one grammatical tense pattern, one suffix, or one naming convention. The event should use recognizable domain terminology, avoid purely technical or generic naming when domain meaning is available, and avoid CRUD-centered or persistence-centered naming when the domain has a more specific concept.

Evaluate relevance conservatively. Not every state change deserves a Domain Event. Event creation for every setter, trivial mutation, method call, persistence operation, or internal technical change can indicate weak semantics when the event is claimed as a Domain Event. Do not require a minimum or maximum number of events.

Evaluate payload semantics in relation to the fact. The event should contain information coherent with the represented occurrence, such as relevant identities, time of occurrence when architecturally necessary, changed values or state needed for legitimate interpretation, and contextual data needed by valid consumers. Do not require the event to contain the entire entity state, and do not require a universal minimum payload.

Evaluate conceptual immutability only as far as needed for this rule. A completed fact should not change meaning after creation, be reused for different facts, be incrementally filled in a way that changes what occurred, or depend on current mutable external state to be understood. Detailed implementation and contract immutability belong to `MSG-005`.

Evaluate the creation moment. A Domain Event should be created after the necessary business decision, validation, and logical transition have occurred, at the point where the fact can be considered true inside the model. Creation before validation, before an operation can still fail, without an actual change, or as a way to request the change is evidence against valid Domain Event semantics.

Keep the boundary with integration events explicit. This rule evaluates the internal domain meaning of a completed fact. It does not evaluate whether an externally published event is a stable integration contract, whether an internal model leaks across boundaries, or whether external consumers can depend on contract evolution. Those concerns belong to `MSG-002`, `MSG-007`, `MSG-008`, and `MSG-009` when applicable.

Keep the boundary with producer reliability explicit. This rule does not evaluate whether state changes and publication intent are durable or consistent under failure, whether a publication record exists, whether delivery is reliable, whether consumers are idempotent, or whether replay is safe. Those concerns belong to adjacent messaging rules such as `MSG-010`, `MSG-011`, `MSG-012`, `MSG-013`, `MSG-014`, and `MSG-020`.

Do not re-evaluate Domain-Driven Design, Fowler Patterns, Hexagonal Architecture, Clean Architecture, Layered Architecture, SOLID, or Architecture Testing. Domain modeling evidence, DTO shape, adapter placement, layer placement, dependency direction, object-oriented design, or tests may help interpret semantics, but findings from this rule must be anchored to whether the event represents a completed domain fact.

Common finding shapes include a command modeled as a Domain Event, an event created before the represented fact is concluded, an event named as success while representing only an attempt, a technical notification classified as a Domain Event, a generic event without domain semantics, an event emitted for trivial changes, payload unrelated to the represented fact, divergence between event name and producer behavior, an event that merely copies an entity, one event type reused for different facts, an event emitted without a domain change, and an event whose interpretation depends on mutable external state.

Remediation guidance should follow the evidence. Suitable remediation may include renaming the event to reflect the completed fact, separating command/request objects from event objects, creating the event only after the relevant decision or transition is valid, aligning payload with the represented fact, separating technical events from domain events, reducing generic wrappers, removing events without domain relevance, separating semantically different events, preserving stable fact meaning, documenting the fact represented, and aligning tests with the correct occurrence moment. Do not prescribe a framework, broker, outbox, event sourcing, aggregate-root-only event production, or architecture testing tool as remediation for this rule.

# Pass Conditions

Use `Pass` when the reviewed evidence is sufficient to identify the Domain Event role and supports that the event represents a completed business fact that has occurred in the domain.

Use `Pass` when the event has clear domain meaning, is tied to a relevant occurrence, decision, rule, process, state transition, internal reaction, integration need, semantic audit, projection, meaningful notification, or consistency concern, and uses language coherent with the reviewed domain.

Use `Pass` when the event is created after the necessary decision, validation, or logical transition; does not function as a command or technical request; and contains information compatible with the represented fact without requiring the entire entity state or a universal payload shape.

Use `Pass` when the event's meaning remains stable after creation and does not depend on a messaging technology, broker, topic, queue, serializer, envelope, or mutable external state to become meaningful.

# Fail Conditions

Use `Fail` only when direct evidence shows that the reviewed element is adopted, declared, or depended on as a Domain Event and violates the semantic responsibility of this rule with real architectural impact.

Use `Fail` when a claimed Domain Event represents an intention, command, request, persistence signal, transport concern, framework callback, technical operation, generic wrapper, internal DTO, or implementation workflow rather than a completed domain-significant fact.

Use `Fail` when direct evidence shows that the event declares a fact that has not occurred, is created before required validation or before the operation can still fail, is emitted to request the same change it claims has happened, is created even when no relevant domain change occurred, or is named as completed success while representing only an attempt.

Use `Fail` when the event has no discernible business semantics, when the same event type is reused for materially different domain facts, when its payload prevents consumers from understanding the represented fact, or when its interpretation depends on mutable external state in a way that undermines the stability of the fact.

Do not use `Fail` for legitimate absence of Domain Events, for technical events correctly classified outside the domain-event scope, for imperfect naming where the completed fact is still clear, or for missing publication, delivery, idempotency, ordering, replay, versioning, or observability mechanisms that belong to other rules.

# Warning Conditions

Use `Warning` when evidence indicates partially clear domain-event semantics but does not justify a definitive `Fail`.

Use `Warning` when an event mixes fact and command language, has excessive genericity, has doubtful relevance, has partially incoherent payload, has divergence between name and behavior, is valid in one producer flow but semantically invalid or premature in another, or relies on strong but incomplete evidence to establish the represented fact.

Use `Warning` when the event appears domain-oriented but is produced around technical lifecycle points, persistence operations, or broad state changes in a way that may weaken the domain meaning, and the available material is not strong enough to prove a confirmed semantic violation.

Use `Warning` when tests, consumers, documentation, or producer code partially support domain semantics but leave important ambiguity about creation timing, factual completion, relevance, payload meaning, or consumer interpretation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope contains no Domain Events, the system does not adopt the Domain Event concept, or the absence of Domain Events is intentional and valid for the reviewed architecture.

Use `Not Applicable` when the reviewed element is explicitly a command, request, integration event, technical event, transport envelope, log, audit record, callback, DTO, persistence signal, or operational notification and is not presented or used as a Domain Event.

Use `Not Applicable` when the reviewed material contains messages or asynchronous infrastructure but no element whose role is to represent a domain-significant fact inside the domain.

Use `Not Applicable` when the scope is limited to publication reliability, delivery behavior, retries, dead-letter handling, idempotency, ordering, replay, contract versioning, schema evolution, observability, adapter placement, layer dependency, or test strategy without a domain-event semantic concern.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the reviewed material does not provide enough evidence to identify the event role, creation point, associated behavior, preceding validation, state transition, represented fact, payload meaning, consumers, documentation, or complete flow.

Use `Not Enough Evidence` when names, suffixes, folders, marker interfaces, comments, diagrams, or isolated schemas suggest Domain Event semantics but there is no traceable behavior showing whether a completed domain fact occurred.

Use `Not Enough Evidence` when implementation and documentation conflict and the reviewed material cannot resolve which source reflects actual event behavior.

Use `Not Enough Evidence` when only producer code or only consumer code is available and the missing side is necessary to determine whether the event is a completed fact, a command, a technical notification, or an integration contract outside this rule's scope.

Use `Not Enough Evidence` when the payload is too generic, opaque, or externally defined to determine its domain meaning and no corroborating tests, documentation, or behavior are available.

# Severity Guidance

Assign severity from demonstrated architectural impact within the reviewed scope. Do not assign fixed severity for this rule, do not use numeric scoring, and do not map `Fail` automatically to high severity.

Severity increases when invalid Domain Event semantics affect critical business flows, central domain decisions, many producers or consumers, important projections, semantic audit trails, integration reactions, consistency decisions, operational behavior, or broadly reused event contracts.

Severity also increases when a premature or false event can cause consumers to make incorrect decisions, propagate information that is not true, create tight coupling to producer implementation details, obscure domain language, or require difficult changes across multiple modules, contexts, services, or retained messages.

Severity may be medium or low when the issue is localized, internal, infrequent, easy to rename or separate, has limited consumers, has limited business effect, or creates maintainability risk without immediate consistency or operational risk.

Informational severity may be appropriate when the reviewed evidence shows only minor naming imperfection or documentation ambiguity and the domain fact remains clear enough that no architectural risk is demonstrated.

# Confidence Guidance

Use `Confirmed` when direct, traceable evidence establishes the element's Domain Event role, the producer flow, the preceding decision or validation, the associated state transition or occurrence, the payload meaning, and whether the event represents a completed domain-significant fact.

Use `Likely` when multiple consistent evidence points strongly support the evaluation but part of the flow, consumer behavior, tests, or documentation is unavailable.

Use `Possible` when evidence is limited, indirect, naming-based with partial corroboration, structurally suggestive, or sufficient only to identify a potential semantic risk.

Use `Not Enough Evidence` when essential information is missing, the flow is not traceable, documentation conflicts with implementation, consumer interpretation is unknown and necessary, or the material cannot distinguish a Domain Event from a command, technical event, integration message, DTO, or generic message.

Do not confuse confidence with severity or outcome. A low-severity issue may be `Confirmed`, and a serious suspected issue may remain `Possible` or `Not Enough Evidence` when the evidence is incomplete. Naming alone must not produce `Confirmed` confidence.
