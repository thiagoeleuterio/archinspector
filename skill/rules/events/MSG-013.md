# Rule ID

MSG-013

# Title

Consumer idempotency

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether repeated processing of the same message or business intent avoids incorrect or duplicated effects when repetition can occur.

# Applicability

Apply this rule when consumers may receive redeliveries, retries, duplicate publications, replayed messages, uncertain acknowledgements, or repeated business intents and perform state changes, external effects, notifications, payments, commands, or other non-trivial side effects.

Do not apply this rule when repeated processing cannot occur in the reviewed context, when the consumer has no side effects or relevant state change, or when duplicate effects are explicitly harmless and evidenced.

# Rule Statement

When repeated processing is possible and effects matter, consumers must make processing idempotent at the level of the message or business intent so repeated attempts do not create incorrect state or duplicated external effects.

# Rationale

At-least-once delivery, retries, partial failures, uncertain commits, and replay can execute the same consumer logic more than once. Idempotency protects business effects; it is not equivalent to exactly-once delivery and is not reducible to storing a message ID.

# Evidence Required

Evaluation requires evidence of consumer handler behavior, side effects, state changes, external calls, transaction boundaries, acknowledgement timing, retry and redelivery paths, idempotency keys, deduplication records, natural idempotent operations, tests, and retention windows.

Primary evidence includes executable consumer code, persistence of processed intent, unique business keys, transaction handling, external effect guards, failure tests, integration tests, and operational records.

# Evaluation Guidance

Evaluate the effect safety of repeated processing. Determine whether the relevant duplicate is the same transport message, same business intent, or same external effect. Inspect atomicity between the effect and any processed marker.

Common finding shapes include charging twice after redelivery, sending repeated irreversible notifications, marking processed before the effect commits, relying only on broker guarantees, deduplicating message ID while business intent repeats under different IDs, and multiple consumer instances racing without atomic guard.

Remediation may include using business idempotency keys, natural idempotent operations, transactional processed records, external idempotency support, effect-before-marker alignment, retention windows appropriate to duplicate risk, or documented acceptance of harmless repetition. Do not require a deduplication table or exactly-once delivery.

Boundaries: `MSG-012` describes delivery semantics; this rule evaluates repeated effect safety. `MSG-014` evaluates duplicate recognition and handling. `MSG-020` evaluates replay safety beyond ordinary repeated processing.

# Pass Conditions

Use `Pass` when evidence shows repeated processing that can occur is harmless, naturally idempotent, or controlled by idempotency mechanisms appropriate to the business intent and side effects.

# Fail Conditions

Use `Fail` when direct evidence shows possible or observed repeated processing can create incorrect state, duplicate external effects, or partial transaction inconsistencies with real impact.

# Warning Conditions

Use `Warning` when idempotency exists but is incomplete, limited to message IDs, weak under concurrency, insufficiently retained, or not demonstrated for uncertain failure paths.

# Not Applicable Conditions

Use `Not Applicable` when redelivery, retry, replay, or repeated intent cannot occur in the reviewed scope, or when consumer processing has no relevant side effects.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when material does not show consumer side effects, retry/redelivery behavior, transaction boundaries, idempotency controls, duplicate keys, or tests needed to evaluate repeated processing.

# Severity Guidance

Severity depends on duplicated effect impact, external irreversibility, financial or regulatory risk, state corruption, concurrency, detection capability, recovery cost, and affected consumer scope.

# Confidence Guidance

Use `Confirmed` when direct evidence shows repeated processing behavior and idempotency result. Use `Likely` when multiple sources support the conclusion with partial failure evidence missing. Use `Possible` for limited signals. Use `Not Enough Evidence` when effect safety cannot be inspected. Message ID presence alone must not produce `Confirmed`.

# Dependencies

MSG-012, MSG-014, MSG-020.
