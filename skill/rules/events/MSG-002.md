# Rule ID

MSG-002

# Title

Integration event boundary

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether messages exposed across architectural boundaries are stable integration contracts rather than accidental exposure of internal models.

# Applicability

Apply this rule when reviewed material includes messages, events, contracts, payloads, schemas, topics, queues, endpoints, files, streams, or other exchanged structures that cross a component, module, service, process, bounded context, organizational, or independently deployable boundary.

Do not apply this rule merely because a system contains internal Domain Events, DTOs, callbacks, notifications, or message-like objects that do not cross a boundary or are not depended on by independent consumers. The absence of integration events is legitimate when the reviewed architecture uses synchronous calls, internal-only events, local coordination, or no messaging for the reviewed scope.

# Rule Statement

Messages exposed across architectural boundaries must be intentionally defined as integration contracts with meaning stable enough for independent consumers, without accidental dependence on mutable internal domain, persistence, or implementation structures.

# Rationale

Integration messages create contracts between independently changing parts of a system. When a producer exposes internal entities, persistence records, object graphs, ORM details, technical wrappers, or domain internals as the external message contract, consumers become coupled to producer implementation choices rather than to a stable integration meaning.

This rule does not require every Domain Event to become an Integration Event, does not require Domain-Driven Design, and does not require distinct types when evidence shows that the same type is intentionally suitable and stable for both internal and external use.

# Evidence Required

Evaluation requires traceable evidence identifying the boundary crossed, the producer, the exposed message contract, the internal model or source data, and one or more consumer or downstream dependency points when available.

Primary evidence includes producer mapping code, serializers, schema definitions, published contracts, endpoint or topic contracts, consumer deserialization and usage, integration tests, contract tests, ADRs, documentation tied to the implementation, and deployment or ownership boundaries. Supporting evidence includes names, folders, suffixes, annotations, routing configuration, and comments, but these signals are not sufficient by themselves for `Confirmed` confidence.

# Evaluation Guidance

First establish that a boundary exists and that the message is used as an integration contract. Then evaluate whether the contract is deliberately shaped for integration, whether consumers can interpret it without depending on internal classes or persistence structure, and whether its meaning remains stable when internal implementation changes.

Common finding shapes include publishing internal entity types directly, leaking persistence identifiers or storage-only fields as required contract semantics, exposing object graphs that force consumers to know producer internals, using a Domain Event externally without evidence that its external meaning was designed, and shared libraries that make consumers compile against producer internals.

Remediation should preserve the integration meaning while reducing accidental coupling. Appropriate guidance may include introducing an explicit contract, mapping from internal state or Domain Event to integration payload, documenting consumer-relevant meaning, removing storage-only details from the contract, or clarifying that an existing shared type is intentionally stable for both roles.

Keep boundaries explicit. `MSG-001` evaluates internal Domain Event fact semantics; this rule evaluates contract exposure across boundaries. `MSG-003` evaluates ownership. `MSG-007`, `MSG-008`, and `MSG-009` evaluate versioning, compatibility, and schema evolution. Do not re-evaluate DDD bounded contexts, Fowler DTO correctness, Hexagonal adapters, Clean boundary data, or Layered placement as the central conclusion.

# Pass Conditions

Use `Pass` when evidence shows that boundary-crossing messages are intentionally shaped as integration contracts, hide internal implementation and persistence details unless those details are legitimate contract semantics, and provide stable meaning for independent consumers.

# Fail Conditions

Use `Fail` when direct evidence shows that a boundary-crossing message exposes internal entities, persistence structures, mutable implementation details, or internal class dependencies as the external contract in a way that creates real coupling or consumer instability.

# Warning Conditions

Use `Warning` when evidence shows partial separation, inconsistent mapping, unclear contract intent, or consumer coupling risk, but the material does not prove a definitive boundary violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope contains no boundary-crossing messages or when the message-like elements are internal, technical, synchronous-only, or otherwise outside integration contract scope.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may apply but the reviewed material does not show the boundary, actual published shape, producer mapping, consumer usage, contract definition, or relationship between internal model and exposed message.

# Severity Guidance

Assign severity from impact on consumer autonomy, number and criticality of consumers, breadth of exposed internals, difficulty of changing producer internals, operational or financial risk, and recoverability. Do not assign severity solely from the presence of shared types or event naming.

# Confidence Guidance

Use `Confirmed` when direct evidence shows the boundary, exposed contract, internal model relationship, and consumer dependence or independence. Use `Likely` when multiple consistent sources support the conclusion with some missing detail. Use `Possible` for limited or indirect evidence. Use `Not Enough Evidence` when the boundary or contract behavior cannot be established. Naming alone must not produce `Confirmed`.

# Dependencies

MSG-001, MSG-003, MSG-007, MSG-008, MSG-009. These are adjacent responsibilities, not procedural prerequisites.
