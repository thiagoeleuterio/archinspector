# Rule ID

MSG-006

# Title

Event naming and meaning

# Category

Events and Messaging

# Status

Active

# Intent

Evaluate whether an event or message name communicates its meaning, occurrence, and observable behavior accurately enough for producers and consumers to interpret it consistently.

# Applicability

Apply this rule when reviewed material includes named events, messages, schemas, contracts, channels, handlers, or documentation where the name is used to convey architectural or business meaning.

Do not apply this rule to unnamed internal payload fragments, transport-only identifiers with no semantic role, or scopes where message naming cannot be inspected. Do not require a specific language, suffix, tense, grammar, or naming convention.

# Rule Statement

An event or message name must align with the behavior and meaning evidenced by its producer, payload, consumers, and documentation, without relying on ambiguous, misleading, purely technical, or contradictory naming as the primary semantic contract.

# Rationale

Names are not proof, but they shape how producers and consumers understand messages. Misleading names can cause consumers to react to attempts as successes, commands as facts, technical changes as business events, or one contract as several different meanings.

# Evidence Required

Evaluation requires evidence of the name, producer behavior, payload, consumer behavior, tests, schemas, documentation, and any channel or type names that carry message meaning.

Primary evidence includes creation flow, state transition, handler behavior, consumer interpretation, contract fields, tests, and docs. Naming itself is relevant evidence for this rule but still must be interpreted with behavior and contract evidence.

# Evaluation Guidance

Evaluate name-to-meaning alignment, not general domain model language quality. A grammatically imperfect or unconventional name should not produce a finding when evidence shows consistent interpretation. A polished name can still fail when behavior contradicts it.

Common finding shapes include generic names such as changed or event, technical names used for business facts, success names used for attempts, command-like names for facts, same name used for different meanings, CRUD names hiding a more specific domain occurrence, semantic meaning hidden only in metadata, and producer/consumer naming divergence.

Remediation may include renaming to match the observed fact or message responsibility, separating command and event names, documenting the contract meaning, aligning producer and consumer terminology, or splitting distinct meanings. Do not prescribe a naming language, suffix, tense, or universal grammar.

Boundaries: `MSG-001` evaluates full Domain Event semantics; this rule evaluates naming alignment. `MSG-004` evaluates granularity. DDD owns broader ubiquitous language consistency.

# Pass Conditions

Use `Pass` when evidence shows that the event or message name accurately communicates the represented occurrence or responsibility and is interpreted consistently by producer, consumer, payload, and documentation.

# Fail Conditions

Use `Fail` when direct evidence shows a misleading or reused name causes real semantic confusion, consumer misinterpretation, command/fact inversion, attempt/success confusion, or contract ambiguity.

# Warning Conditions

Use `Warning` when naming is generic, technical, inconsistent, or weakly aligned with behavior, but consumer impact or semantic contradiction is partial or not fully proven.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed material contains no named event or message element whose name communicates architecture or business meaning.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when names are visible but producer behavior, consumer interpretation, payload meaning, or documentation is insufficient to evaluate alignment.

# Severity Guidance

Severity depends on criticality of affected flows, consumer reliance on the name, risk of wrong reactions, breadth of reuse, external exposure, retained-message impact, and correction difficulty. Do not assign severity from naming aesthetics.

# Confidence Guidance

Use `Confirmed` when direct evidence shows the name and corroborating behavior or contradiction. Use `Likely` when several signals align but one side is incomplete. Use `Possible` for weak naming concerns with partial support. Use `Not Enough Evidence` when meaning cannot be traced. Naming alone should not produce `Confirmed` failure.
