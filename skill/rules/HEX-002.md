# Rule ID

HEX-002

# Title

Application core must expose inbound ports

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether external actors enter the application core through explicit application-facing boundaries within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes application core behavior and one or more external actors, delivery mechanisms, or inbound interaction paths that invoke that behavior. The rule may be considered whether inbound boundaries are explicitly named as ports or represented by equivalent application-facing contracts.

# Rule Statement

The application core must expose inbound ports for external actors to invoke core behavior.

# Rationale

Explicit inbound ports make the entry points into core behavior visible, reduce coupling between delivery mechanisms and application behavior, and help keep external interaction concerns from shaping the internal structure of the core.

# Evidence Required

Evaluation requires traceable evidence that identifies the application core scope and the reviewed inbound interaction paths. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, or reviewed code excerpts showing how external actors or inbound adapters invoke core behavior. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between the application core and inbound actors or adapters. Then evaluate whether inbound interaction paths target explicit application-facing boundaries rather than arbitrary internal implementation details. Interpret evidence conservatively, and do not infer the existence of inbound ports from naming, intended architecture, or unavailable material. An inbound boundary should be considered relevant when it represents a stable entry point into core behavior within the reviewed scope.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant inbound interaction paths and the available evidence supports that external actors invoke core behavior through explicit inbound ports.

# Fail Conditions

Use `Fail` when direct evidence shows that external actors or inbound adapters invoke core behavior through internal implementation details, bypass explicit application-facing boundaries, or require delivery mechanisms to coordinate core behavior without an inbound port.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, implicit, or ambiguous inbound boundaries, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include application core behavior, external actors, delivery mechanisms, or inbound interaction paths relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the application core scope, identify inbound interaction paths, inspect how core behavior is invoked, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when missing or bypassed inbound ports affect central application behavior, broad inbound surfaces, or stable architectural boundaries. Assign lower severity when the issue is narrow, isolated, or limited to peripheral interaction paths. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant scopes, inbound interaction paths, and boundary shape. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.
