# Rule ID

HEX-012

# Title

Core behavior must be testable without adapters

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether application core behavior can be evaluated independently of infrastructure and delivery mechanisms within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes application core behavior and one or more adapters, delivery mechanisms, infrastructure mechanisms, or external integration components that interact with that behavior. The rule is relevant only when there is enough context to inspect whether core behavior can be exercised without requiring those external-facing mechanisms.

# Rule Statement

Core behavior must be testable without adapters.

# Rationale

Core behavior that cannot be evaluated without adapters is coupled to delivery or infrastructure mechanisms. This weakens core isolation, makes behavior harder to verify independently, and can hide architectural boundary problems behind external setup or technical integration requirements.

# Evidence Required

Evaluation requires traceable evidence that identifies the application core scope, the reviewed core behavior, and the adapters, delivery mechanisms, infrastructure mechanisms, or external integration components related to that behavior. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, configuration dependencies, or reviewed code excerpts showing whether core behavior can be instantiated, invoked, or reasoned about through core contracts without concrete adapters. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between application core behavior and adapters, delivery mechanisms, or infrastructure mechanisms. Then evaluate whether core behavior depends on concrete adapters or external-facing mechanisms to execute its application decisions, or whether it can be exercised through core contracts and substitutable boundaries. Interpret evidence conservatively, and do not infer testability from the mere presence or absence of tests. This rule evaluates independence of core behavior from adapters for evaluation purposes; it does not evaluate testing strategy, coverage, scoring, framework leakage, persistence leakage, messaging leakage, or general dependency direction except as evidence for this specific condition.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant core behavior and the available evidence supports that the behavior can be evaluated through core contracts without requiring concrete adapters, delivery mechanisms, infrastructure mechanisms, or external integration components.

# Fail Conditions

Use `Fail` when direct evidence shows that core behavior requires concrete adapters, delivery mechanisms, infrastructure mechanisms, external integration components, or adapter-specific setup to execute or evaluate the core behavior within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, or ambiguous dependence on adapters for evaluating core behavior, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include application core behavior, adapters, delivery mechanisms, infrastructure mechanisms, external integration components, or evaluable core behavior relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the application core scope, identify core behavior, identify related adapters or external-facing mechanisms, inspect whether core behavior can be exercised without those mechanisms, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when adapter-dependent evaluation affects central application behavior, broad core behavior surfaces, multiple adapters, or stable architectural boundaries. Assign lower severity when the dependence is narrow, isolated, indirect, or limited to peripheral behavior. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant core scope, core behavior, related adapters or external-facing mechanisms, and whether core behavior can be evaluated without them. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies
