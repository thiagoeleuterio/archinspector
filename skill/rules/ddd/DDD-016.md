# Rule ID

DDD-016

# Title

Shared kernels must remain explicit and limited

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure shared domain model elements are intentionally bounded and do not create uncontrolled coupling between contexts.

# Applicability

Apply this rule when the reviewed material includes domain model elements, language, contracts, rules, or code that appear to be shared between bounded contexts or candidate contexts as a Shared Kernel or equivalent shared domain model responsibility. The rule is relevant only when the reviewed scope provides enough evidence to inspect whether the sharing is intentional, explicit, limited, and governed by the participating contexts.

# Rule Statement

Shared kernels must be explicit, intentionally limited, and governed by the bounded contexts that share them.

# Rationale

A Shared Kernel can reduce duplication where participating contexts intentionally share a small part of a domain model, but it also creates coupling between those contexts. When shared domain elements are implicit, broad, or unmanaged, changes in one context can unexpectedly affect another, model ownership becomes unclear, and shared structures can expand beyond the domain meaning they were meant to preserve.

# Evidence Required

Evaluation requires traceable evidence that identifies the participating contexts, the shared domain model elements, the intended scope of sharing, and the governance or coordination mechanism for changes to the shared elements. Direct evidence may include shared contracts, shared modules or packages with domain model content, dependency relationships, type definitions, domain rule definitions, ownership documentation, change coordination documentation, reviewed code excerpts, or documentation tied to the involved contexts.

Evidence must distinguish a Shared Kernel from ordinary shared utilities, platform libraries, framework abstractions, technical helper code, generated contracts, persistence models, transport models, generic common packages, or duplicated code. Naming conventions, shared folder names, common package names, or kernel labels may support interpretation, but they must not be treated as conclusive evidence without corroborating domain-model, ownership, boundary, dependency, or documentation evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the shared material represents domain model elements shared between bounded contexts rather than generic technical reuse. Then inspect whether that shared kernel is explicit, intentionally limited to a small and stable model area, and governed by the participating contexts so that ownership and change impact can be identified.

Evaluate only Shared Kernel responsibility. Do not treat any shared library, shared project, shared package, or common type as a Shared Kernel without evidence that it carries domain model meaning shared by participating contexts. Do not conclude a violation solely from the existence of shared code. Do not use this rule to evaluate general module reuse, solution structure, dependency direction, adapter contracts, Clean Architecture boundary data, Hexagonal ports, or bounded context internal boundaries unless that evidence directly shows whether a Shared Kernel is explicit, limited, and governed.

Interpret evidence conservatively. A broad shared module may be acceptable when the reviewed material shows the shared part is not domain model content relevant to a Shared Kernel. A finding requires evidence that domain model elements are shared across contexts and that the sharing is implicit, excessive, uncontrolled, or lacks visible governance within the reviewed scope.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify the participating contexts and shared domain model elements, and the available evidence supports that the Shared Kernel is explicit, intentionally limited, and governed by the contexts that share it.

# Fail Conditions

Use `Fail` when direct evidence shows that shared domain model elements between bounded contexts are implicit, broad enough to blur context ownership, changed without visible shared governance, used as an uncontrolled coupling mechanism, or treated as a general common model across contexts within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, emerging, or ambiguous Shared Kernel explicitness, limitation, or governance, but the available material does not justify a definitive `Fail`. Use `Warning` when sharing appears intentional but its scope or governance is only partially visible.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include domain model elements shared between bounded contexts, candidate Shared Kernels, equivalent shared domain model responsibilities, or Shared Kernel concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify participating contexts, distinguish shared domain model elements from technical reuse, inspect the intended scope of sharing, determine whether the sharing is governed, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when uncontrolled Shared Kernel coupling affects central domain concepts, critical shared rules, multiple bounded contexts, broad model surfaces, or stable context boundaries. Assign lower severity when the shared domain element is narrow, isolated, peripheral, explicitly temporary, or limited to low-impact model content. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the participating contexts, the shared domain model elements, the intended shared scope, and whether the Shared Kernel is explicit, limited, and governed. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming or shared physical location alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002, DDD-003, DDD-014. These are conceptual dependencies for shared domain language, bounded context, context relationship, and shared model vocabulary, not procedural prerequisites for evaluating this rule.
