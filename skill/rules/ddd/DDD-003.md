# Rule ID

DDD-003

# Title

Bounded contexts must define explicit model boundaries

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure distinct domain models have identifiable boundaries and do not blur incompatible meanings.

# Applicability

Apply this rule when the reviewed material includes multiple domain areas, domain models, subdomains, business capabilities, domain-facing contracts, context relationships, or terminology that may carry different meanings in different parts of the system. The rule is relevant only when the reviewed scope provides enough evidence to inspect whether distinct domain models have explicit boundaries.

# Rule Statement

Bounded contexts must define explicit model boundaries that keep distinct domain meanings identifiable.

# Rationale

Bounded contexts protect the integrity of domain models by making the boundary of each model's language, rules, and meaning visible. When context boundaries are implicit, missing, or blurred, incompatible meanings can leak across the system, domain rules can be applied outside their intended model, and reviewers have weaker evidence for deciding which model owns a concept or behavior.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed bounded contexts or candidate context boundaries, the domain models or meanings they contain, and the relationships or interactions between them. Direct evidence may include project or module references, namespace relationships, type dependencies, domain-facing contracts, context-specific terminology, model definitions, boundary abstractions, translation behavior, method behavior, state transition behavior, reviewed code excerpts, or documentation tied to the reviewed domain models.

Evidence must distinguish bounded contexts from physical deployment units, modules, projects, packages, microservices, repositories, folders, namespaces, teams, or databases. These structures may support interpretation when corroborated by model, language, boundary, behavior, or relationship evidence, but they must not be treated as sufficient evidence of bounded context integrity by themselves.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that distinct domain meanings or domain models exist within the reviewed scope. Then inspect whether each model boundary is explicit enough to identify where its language and rules apply, how neighboring models interact, and whether incompatible meanings are separated or translated rather than silently shared.

Evaluate only the existence and integrity of bounded context model boundaries. Do not use this rule to evaluate physical architecture, module quality, microservice decomposition, solution structure, source dependency direction, ports and adapters, Clean Architecture use case boundaries, aggregate consistency boundaries, aggregate root access control, context relationship types, anti-corruption layers, shared kernels, published languages, upstream or downstream roles, or domain isolation unless that evidence directly shows whether bounded context model boundaries are explicit and intact.

Interpret evidence conservatively. A project, namespace, service, or folder named after a business area is not automatically a bounded context. A shared type or shared term is not automatically a violation unless evidence shows that it blurs incompatible domain meanings or prevents the boundary of a model from being identified. Prefer `Not Enough Evidence` when the reviewed material does not allow model boundaries to be distinguished from physical organization or naming intent.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant distinct domain models or meanings, and the available evidence supports that bounded context boundaries are explicit enough to show where each model's language, rules, and meaning apply and how neighboring models interact without blurring incompatible meanings.

# Fail Conditions

Use `Fail` when direct evidence shows that distinct domain models or incompatible domain meanings are mixed without an identifiable boundary, that domain rules from one model are applied as if they belonged to another model, that shared structures force incompatible meanings into a single model, or that interactions between models bypass any visible boundary needed to preserve meaning within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, implicit, emerging, or ambiguous bounded context boundaries, but the available material does not justify a definitive `Fail`. Use `Warning` when boundaries are visible in some areas but unclear in others, or when physical organization suggests separation while model language or behavior provides incomplete corroboration.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include multiple domain areas, distinct domain models, context relationships, incompatible meanings, or bounded context concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify distinct domain models, inspect context boundaries, distinguish bounded contexts from physical organization, determine whether meanings are compatible or incompatible, inspect interactions between candidate contexts, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when blurred bounded context boundaries affect central domain models, broad cross-context behavior, critical business meanings, many integrations, or the ability to reason about rule ownership. Assign lower severity when boundary ambiguity is narrow, isolated, peripheral, or limited to a small model area with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the distinct domain models or meanings, the relevant model boundaries, and whether those boundaries are explicit and intact. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming or physical structure alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002. This is a conceptual dependency for shared domain language vocabulary, not a procedural prerequisite for evaluating this rule.
