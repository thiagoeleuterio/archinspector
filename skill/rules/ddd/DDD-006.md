# Rule ID

DDD-006

# Title

Entities must preserve identity and lifecycle consistency

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure entities represent domain identity and lifecycle behavior consistently.

# Applicability

Apply this rule when the reviewed material includes identifiable entities, entity-like domain concepts, lifecycle identity, state transitions, lifecycle rules, or behavior associated with preserving continuity of a domain object over time. The rule is relevant only when the reviewed scope provides enough evidence to inspect entity identity, lifecycle behavior, and state consistency for the entity being evaluated.

# Rule Statement

Entities must preserve domain identity and lifecycle consistency across their state transitions.

# Rationale

Entities express domain concepts whose meaning depends on identity that continues through lifecycle changes. When an entity does not preserve identity consistently, callers may treat distinct lifecycle objects as interchangeable, lose track of continuity over time, or coordinate lifecycle transitions outside the model. When lifecycle behavior is inconsistent or scattered, the domain model becomes harder to reason about and entity state may no longer reflect valid domain progression.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed entity or entity-like domain concept, the identity that distinguishes it over time, and the lifecycle behavior or state transitions associated with that identity. Direct evidence may include type definitions, member definitions, identity member usage, equality or comparison behavior, construction paths, method behavior, state transition behavior, lifecycle rule checks, dependency relationships involving the entity, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish entities from value objects, aggregates, aggregate roots, domain services, application services, repositories, data transfer structures, persistence structures, serialization structures, framework models, and technical identifiers. Naming conventions, entity labels, folder names, or identifier-like members may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the concept being evaluated is an entity or entity-like domain concept whose meaning depends on lifecycle identity rather than value identity. Then inspect whether the entity preserves that identity consistently across construction, equality or comparison when relevant, state transitions, lifecycle changes, and behavior that represents domain progression.

Evaluate only entity identity and lifecycle consistency. Do not use this rule to evaluate value object invariant protection or value-based identity, aggregate consistency boundary enforcement, Aggregate Root access control, domain service behavior placement, application service coordination, repository abstraction, Clean Architecture entity independence from use cases or adapters, source dependency direction, ports and adapters, persistence mapping, serialization behavior, or framework-specific implementation details unless that evidence directly shows whether the entity preserves identity or lifecycle consistency.

Interpret evidence conservatively. An object with an identifier is not automatically an entity, and an entity is not required to expose arbitrary lifecycle behavior. A finding requires evidence that lifecycle identity or lifecycle behavior belongs to the entity and is missing, contradicted, bypassed, duplicated elsewhere, or inconsistently enforced. Technical identifiers should be evaluated only when the reviewed domain model shows that they represent or interfere with domain identity.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant entities, their domain identity, and their lifecycle behavior, and the available evidence supports that those entities preserve identity and lifecycle consistency across relevant state transitions.

# Fail Conditions

Use `Fail` when direct evidence shows that an entity loses, changes, contradicts, or exposes inconsistent domain identity; uses value-based identity where lifecycle identity is required; permits lifecycle transitions that contradict the entity's domain progression; or externalizes lifecycle behavior that the reviewed domain model shows belongs to the entity.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, duplicated, emerging, or ambiguous entity identity or lifecycle consistency, but the available material does not justify a definitive `Fail`. Use `Warning` when identity appears mostly stable but lifecycle transitions, equality behavior, or behavior ownership remain unclear or partially scattered.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable entities, entity-like domain concepts, lifecycle identity, lifecycle behavior, or entity state transitions relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify entities, distinguish entities from value objects or technical structures, inspect identity semantics, inspect lifecycle behavior, determine whether lifecycle transitions are valid, determine whether behavior belongs to the entity, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when weak entity identity or lifecycle consistency affects central domain concepts, broad domain behavior, critical lifecycle transitions, many callers, or stable domain boundaries. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to an entity with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the entity role, the relevant domain identity or lifecycle behavior, and whether the entity preserves or fails to preserve lifecycle consistency. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-001, DDD-002, DDD-003. These are conceptual dependencies for shared value object, domain language, bounded context, entity, and identity vocabulary, not procedural prerequisites for evaluating this rule.
