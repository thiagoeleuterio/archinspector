# Rule ID

DDD-014

# Title

Context relationships must be explicit

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure relationships between bounded contexts are visible enough to support evidence-based review.

# Applicability

Apply this rule when the reviewed material includes multiple bounded contexts, candidate bounded contexts, distinct domain models, inter-context collaboration, cross-context dependencies, shared domain-facing contracts, or interactions where one domain model relates to another. The rule is relevant only when the reviewed scope provides enough evidence to inspect whether relationships between bounded contexts can be identified and reasoned about.

# Rule Statement

Relationships between bounded contexts must be explicit enough to identify dependencies, collaboration, and responsibility boundaries between the participating contexts.

# Rationale

Explicit context relationships help reviewers understand how distinct domain models depend on each other, where collaboration occurs, and which responsibilities remain inside each context. When relationships are implicit or only discoverable through scattered implementation details, incompatible meanings can cross boundaries unnoticed, integration responsibilities become harder to assign, and architectural review has weaker evidence for evaluating context-level risks.

# Evidence Required

Evaluation requires traceable evidence that identifies the participating bounded contexts or candidate contexts and the relationships between them. Direct evidence may include context documentation, module or project relationships, domain-facing contracts, integration contracts, dependency relationships, type dependencies, method behavior, translation behavior, shared abstractions, invocation paths, reviewed code excerpts, or documentation tied to the reviewed domain models.

Evidence must distinguish relationships between bounded contexts from internal bounded context boundaries, ordinary module references, package references, deployment topology, database access, framework integration, generic service calls, or naming conventions. Physical structure, service names, namespace labels, or diagrams may support interpretation, but they must not be treated as conclusive evidence without corroborating model, contract, dependency, behavior, or documentation evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that more than one bounded context or candidate context is relevant within the reviewed scope. Then inspect whether relationships between those contexts are explicit enough to identify which contexts collaborate, which dependencies exist, what responsibility boundary is crossed, and what reviewed material supports that interpretation.

Evaluate only the explicitness of relationships between bounded contexts. Do not use this rule to evaluate whether each bounded context defines its internal model boundary, which belongs to `DDD-003`. Do not use this rule to evaluate anti-corruption layers, shared kernels, published languages, upstream or downstream roles, general source dependency direction, ports and adapters, Clean Architecture use case boundaries, messaging infrastructure, persistence architecture, or solution-level module organization unless that evidence directly shows whether a bounded context relationship is explicit.

Interpret evidence conservatively. A specific Context Map format is not required. A relationship can be explicit through documentation, contracts, dependencies, translation behavior, collaboration paths, or other traceable reviewed material. A type, namespace, service, or folder name is not enough by itself to confirm a bounded context relationship. Prefer `Not Enough Evidence` when relationships are plausible but cannot be traced to reviewed material.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant bounded contexts or candidate contexts and the available evidence supports that their relationships are explicit enough to identify dependencies, collaboration, and responsibility boundaries between the participating contexts.

# Fail Conditions

Use `Fail` when direct evidence shows that bounded contexts collaborate, depend on each other, exchange domain-facing information, or share responsibilities while the relationship between them remains implicit, untraceable, contradictory, or too unclear to identify the dependency, collaboration path, or crossed responsibility boundary within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, emerging, or ambiguous context relationship explicitness, but the available material does not justify a definitive `Fail`. Use `Warning` when some relationships are visible while others are only partially documented or can be inferred from implementation signals without enough corroboration for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include multiple bounded contexts, candidate bounded contexts, distinct domain models, inter-context collaboration, cross-context dependencies, or context relationship concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify participating contexts, inspect relationships between them, distinguish context relationships from physical structure or naming, determine whether dependencies or collaboration cross a context boundary, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when implicit or unclear context relationships affect central bounded contexts, broad cross-context collaboration, critical domain responsibilities, many integrations, or the ability to reason about ownership of domain meaning. Assign lower severity when the relationship ambiguity is narrow, isolated, peripheral, or limited to a small interaction with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the participating contexts, the relationship between them, and whether that relationship is explicit enough to identify dependencies, collaboration, and responsibility boundaries. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming or physical structure alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002, DDD-003. These are conceptual dependencies for shared domain language, bounded context, and model boundary vocabulary, not procedural prerequisites for evaluating this rule.
