# Rule ID

DDD-015

# Title

Anti-corruption layers must protect domain models from external models

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure external or foreign models are translated before they shape the local domain model.

# Applicability

Apply this rule when the reviewed material includes a local bounded context or domain model that interacts with external, foreign, legacy, upstream, partner, third-party, or otherwise non-local models. The rule is relevant only when the reviewed scope provides enough evidence to inspect whether those external models are translated before they influence local domain concepts, rules, or model shape.

# Rule Statement

Anti-corruption layers must translate external or foreign models before those models can shape the local domain model.

# Rationale

An anti-corruption layer protects a local domain model from adopting meanings, structures, assumptions, or constraints that belong to another model. When external models are used directly as local domain concepts, the local model can lose its own language, accept foreign invariants, or become coupled to changes outside its bounded context. Translation preserves the local model's meaning while still allowing collaboration with other contexts or systems.

# Evidence Required

Evaluation requires traceable evidence that identifies the local domain model, the external or foreign model, and the translation boundary or lack of translation between them. Direct evidence may include domain-facing contracts, external contracts, type dependencies, method signatures, mapping or translation behavior, adapter or integration behavior when it shows model translation, construction paths, state transition behavior, reviewed code excerpts, or documentation tied to the involved models.

Evidence must distinguish anti-corruption translation from generic adapter implementation, Clean Architecture interface adapter translation, data mapper behavior, persistence mapping, transport serialization, DTO naming, gateway isolation, and Hexagonal adapter model leakage. Naming conventions, suffixes, folder names, external labels, or anti-corruption terminology may support interpretation, but they must not be treated as conclusive evidence without corroborating structural, behavioral, contract, or documentation evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between the local domain model and an external or foreign model. Then inspect whether the external model is translated into local domain language, concepts, and invariants before it shapes local domain objects, rules, decisions, or state transitions.

Evaluate only protection of the local domain model from external or foreign models through anti-corruption translation. Do not use this rule to evaluate adapters generically. Do not duplicate `HEX-011`, which evaluates adapter model leakage into core contracts. Do not duplicate `CLEAN-006`, `CLEAN-010`, or `CLEAN-011`, which evaluate interface adapter translation, data mapper influence, and boundary data structures under Clean Architecture. Use adapter, mapper, or boundary evidence only when it directly shows whether a foreign model shapes the local domain model.

Interpret evidence conservatively. Direct use of data from another system is not automatically a violation if the reviewed material shows that the data is translated into local domain meaning before affecting the model. Similar fields between external and local structures are not enough by themselves to prove corruption. A finding requires evidence that the local domain model is shaped by external meaning, structure, assumptions, or invariants without a visible translation boundary.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify the local domain model, the external or foreign model, and the interaction between them, and the available evidence supports that external models are translated into local domain meaning before they influence local domain objects, rules, decisions, or state transitions.

# Fail Conditions

Use `Fail` when direct evidence shows that external or foreign models are used as local domain model elements, determine local domain concepts, impose foreign invariants or structures, drive local state transitions, or otherwise shape the local domain model without a visible anti-corruption translation boundary within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, duplicated, or ambiguous protection from external models, but the available material does not justify a definitive `Fail`. Use `Warning` when translation exists for some interaction paths but not others, or when local domain structures appear partly shaped by a foreign model without enough evidence for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include a local domain model interacting with external, foreign, legacy, upstream, partner, third-party, or otherwise non-local models relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the local model, identify the external or foreign model, inspect translation behavior, determine whether external meanings shape local domain concepts, distinguish anti-corruption translation from generic mapping or adapter behavior, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when unprotected external model influence affects central domain concepts, critical invariants, broad context collaboration, many local behaviors, or stable bounded context boundaries. Assign lower severity when the influence is narrow, isolated, peripheral, translated before important domain behavior, or limited to a small model area with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the local model, the external or foreign model, the translation boundary or lack of one, and whether the external model shapes the local domain model. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002, DDD-003, DDD-014. These are conceptual dependencies for shared domain language, bounded context, local model, external model, and context relationship vocabulary, not procedural prerequisites for evaluating this rule.
