# Rule ID

DDD-001

# Title

Value objects should protect invariants

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure value objects preserve domain meaning by protecting invariants.

# Applicability

Apply this rule when the reviewed material includes identifiable value objects, domain value concepts, or domain model structures that are intended to represent descriptive values rather than lifecycle identity. The rule is relevant only when the reviewed scope provides enough evidence to inspect the value object's domain meaning, value-based identity, invariant protection, equality semantics, state transitions, or behavior associated with preserving valid value states.

# Rule Statement

Value objects must preserve domain meaning by protecting their invariants, preventing invalid value states, and representing identity through value rather than lifecycle identity.

# Rationale

Value objects express domain concepts whose meaning comes from their values. When a value object does not protect the invariants that make the value meaningful, invalid states can enter the domain model, equality can become inconsistent with domain meaning, and domain behavior may be scattered into callers that should not own the value's rules. A value object may contain behavior, and behavior that preserves the value object's invariants belongs with the value object when the reviewed domain model shows that the behavior is part of the value concept.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed value object or value-like domain concept and the invariants, equality semantics, state rules, or value-preserving behavior associated with it. Direct evidence may include type definitions, member definitions, construction paths, method behavior, equality behavior, state transition behavior, domain rule checks, reviewed code excerpts, dependency relationships involving the value object, or documentation tied to the reviewed domain model.

Evidence must distinguish value objects from entities, aggregates, aggregate roots, services, data transfer structures, persistence structures, serialization structures, application input validation, and technical boundary models. Naming conventions, suffixes, folder names, or domain labels may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the concept being evaluated is a value object or value-like domain concept. Then inspect whether the value object protects the invariants that preserve its domain meaning, whether invalid value states can be created or persisted inside the domain model, whether equality is based on value rather than lifecycle identity, and whether behavior that preserves those invariants is located with the value object when the domain model assigns that responsibility to the value concept.

Evaluate only value object modeling responsibility. Do not use this rule to evaluate entities, aggregates, aggregate roots, domain services, application services, repositories, factories, domain events, bounded contexts, context mapping, ports and adapters, frameworks, persistence, serialization, data transfer structures, application input validation, or language-specific implementation details unless that evidence directly shows whether the value object protects its own domain invariants.

Interpret evidence conservatively. A value object is not required to contain arbitrary behavior, and an object with little behavior is not automatically invalid. A finding requires evidence that invariant-preserving behavior or state protection belongs to the value object and is missing, bypassed, duplicated elsewhere, or inconsistently enforced. Immutability should be evaluated only when the reviewed domain model shows that stable value semantics or invariant protection require value state not to change after creation or controlled transformation.

Common finding patterns include value-like domain concepts that allow invalid internal state, equality that is inconsistent with value identity, invariant checks scattered across callers, behavior that preserves value meaning placed outside the value object, or mutable value state that can bypass domain rules. These patterns support findings only when corroborated by evidence from the reviewed material.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant value objects and their domain invariants, and the available evidence supports that those value objects preserve domain meaning by protecting invariants, preventing invalid value states, using value-based equality where equality is relevant, and keeping invariant-preserving behavior with the value object when that behavior belongs to the value concept.

# Fail Conditions

Use `Fail` when direct evidence shows that a value object or value-like domain concept can enter invalid value states, exposes state changes that bypass its invariants, uses lifecycle identity where domain meaning depends on value identity, defines equality in a way that contradicts value semantics, or externalizes invariant-preserving behavior that the reviewed domain model shows belongs to the value object.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, duplicated, emerging, or ambiguous protection of value object invariants, but the available material does not justify a definitive `Fail`. Use `Warning` when a value object mostly preserves domain meaning but has unclear equality semantics, partially scattered invariant checks, uncertain mutability risks, or behavior placement that appears inconsistent without enough evidence for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable value objects, value-like domain concepts, domain invariants belonging to values, value-based equality concerns, or value-preserving behavior relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify value objects, distinguish value objects from other domain or technical structures, inspect invariant enforcement, inspect equality semantics, determine whether invalid value states are possible, determine whether behavior belongs to the value object, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when weak value object invariant protection affects central domain concepts, broad domain behavior, critical consistency rules, or many callers that depend on the value's meaning. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to a value concept with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the value object role, the relevant invariant or value semantics, and whether the value object protects or fails to protect that meaning. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-012, DDD-013. These are conceptual dependencies for shared domain invariant and domain model behavior vocabulary, not procedural prerequisites for evaluating this rule.
