# Rule ID

DDD-005

# Title

Aggregate roots must control aggregate access

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure aggregate roots are the controlled entry point for modifying aggregate state.

# Applicability

Apply this rule when the reviewed material includes identifiable aggregates, aggregate roots, aggregate members, related entities or value objects inside an aggregate boundary, or state modification paths that affect aggregate state. The rule is relevant only when the reviewed scope provides enough evidence to inspect how aggregate members are accessed or modified.

# Rule Statement

Aggregate roots must be the controlled entry point for accessing and modifying aggregate state.

# Rationale

Aggregate roots protect aggregate boundaries by controlling how clients reach and change internal aggregate members. When callers modify aggregate members directly, the aggregate boundary becomes weaker, internal structure becomes part of external collaboration, and aggregate state can be changed without passing through the model element responsible for coordinating aggregate access. Controlled access also helps reviewers identify which object represents the aggregate boundary in the reviewed model.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed aggregate, its aggregate root or candidate root, the internal aggregate members, and the access or modification paths used by callers. Direct evidence may include type definitions, member visibility, method signatures, method behavior, state transition behavior, construction paths, dependency relationships involving aggregate members, references from outside the aggregate boundary, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish Aggregate Root access control from aggregate consistency rule enforcement, value object invariant protection, entity lifecycle consistency, repository design, application service orchestration, persistence mapping, serialization behavior, object-relational loading, framework access requirements, module boundaries, ports, adapters, gateways, and Clean Architecture policy boundaries. Naming conventions, root labels, folder names, or object hierarchy labels may support interpretation, but they must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that an aggregate and Aggregate Root can be identified. Then inspect whether callers outside the aggregate boundary interact with aggregate state through the root or whether they can directly access, mutate, replace, or coordinate internal aggregate members in a way that bypasses the root as the controlled entry point.

Evaluate only Aggregate Root control over access and modification of aggregate state. Do not use this rule to re-evaluate whether the aggregate enforces its internal consistency rules, which belongs to `DDD-004`. Do not use this rule to evaluate repository abstraction, persistence loading strategies, application use case isolation, general source dependency direction, entity lifecycle modeling, value object invariants, ports and adapters, or Clean Architecture gateway and boundary responsibilities unless that evidence directly shows whether aggregate access bypasses or respects the Aggregate Root.

Interpret evidence conservatively. The existence of a type named as a root is not enough to prove root control, and visible internal members are not automatically a violation unless evidence shows that callers outside the aggregate boundary can use them to bypass the root's control of aggregate access or modification. A finding requires evidence of the aggregate boundary, the root role, and the relevant access or modification path.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant aggregates, aggregate roots, internal members, and modification paths, and the available evidence supports that callers access and modify aggregate state through the Aggregate Root rather than directly controlling internal aggregate members.

# Fail Conditions

Use `Fail` when direct evidence shows that callers outside the aggregate boundary can directly modify, replace, coordinate, or depend on internal aggregate members in a way that bypasses the Aggregate Root as the controlled entry point for aggregate state within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, or ambiguous Aggregate Root access control, but the available material does not justify a definitive `Fail`. Use `Warning` when some modification paths use the root while others are unclear, or when internal aggregate members appear externally reachable without enough evidence to confirm bypassed modification.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable aggregates, aggregate roots, internal aggregate members, aggregate state modification paths, or Aggregate Root access concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the aggregate, identify the Aggregate Root, distinguish internal members from external collaborators, inspect access or modification paths, determine whether callers bypass the root, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when bypassed Aggregate Root control affects central aggregates, broad aggregate state surfaces, many callers, critical modification paths, or stable domain boundaries. Assign lower severity when the bypass risk is narrow, isolated, read-only, peripheral, or limited to aggregate state with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the aggregate boundary, the Aggregate Root role, the internal aggregate members, and whether access or modification occurs through the root or bypasses it. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-004. This is a conceptual dependency for shared aggregate and consistency boundary vocabulary, not a procedural prerequisite for evaluating this rule.
