# Rule ID

DDD-013

# Title

Domain models must preserve behavioral richness

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure domain models contain meaningful behavior and do not collapse into passive data structures when domain behavior is present.

# Applicability

Apply this rule when the reviewed material includes identifiable domain models, entities, value objects, aggregates, domain services, domain behavior, domain decisions, state transitions, or business rules whose ownership can be inspected. The rule is relevant only when the reviewed scope provides enough evidence to determine whether meaningful domain behavior exists and whether the domain model preserves that behavior.

# Rule Statement

Domain models must preserve meaningful domain behavior when the domain contains decisions, rules, or state transitions that belong to the model.

# Rationale

A domain model should express the behavior that gives domain concepts their meaning when the reviewed domain contains meaningful decisions or transitions. When domain objects become passive data structures while domain decisions are consistently performed elsewhere, the model becomes harder to reason about, domain behavior may be duplicated, and maintainers may need to reconstruct business meaning from external procedures. This rule must remain conservative because some domain concepts are legitimately simple data carriers or descriptive structures with little behavior.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed domain model element, the meaningful domain behavior or decision related to it, and the location where that behavior is expressed or externalized. Direct evidence may include type definitions, member definitions, method behavior, state transition behavior, domain rule checks, construction paths, dependencies involving domain model elements, behavior in services or coordinators, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish behaviorally rich domain models from simple data structures, boundary data structures, persistence structures, serialization structures, transport models, application input or output models, technical DTOs, read models, projections, framework models, and objects without demonstrated behavioral responsibility. Naming conventions, anemic model labels, folder names, or domain suffixes may support interpretation but must not be treated as conclusive evidence without corroborating behavioral or structural evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that meaningful domain behavior, decisions, or state transitions exist and belong to the domain model. Then inspect whether the model elements preserve that behavior or whether the behavior is externalized into callers, application services, procedural coordinators, or technical structures in a way that leaves the domain model passive despite demonstrated behavioral responsibility.

Evaluate only preservation of behavioral richness in the domain model. Do not conclude that every model dominated by data is automatically anemic. Do not require artificial methods, arbitrary encapsulation, or behavior on objects that the reviewed domain model does not show to own behavior. Do not use this rule to duplicate domain service behavior placement in `DDD-007`, application service domain decision ownership in `DDD-008`, mandatory invariant enforcement in `DDD-012`, entity lifecycle consistency in `DDD-006`, value object invariant protection in `DDD-001`, source dependency direction, ports and adapters, or Clean Architecture policy boundaries unless that evidence directly shows whether meaningful domain behavior is preserved by the model.

Interpret evidence conservatively. Public state, simple properties, records, data carriers, or low method counts are not enough by themselves to prove an anemic domain model. A finding requires evidence that meaningful domain behavior belongs to the model and is missing, scattered, duplicated, or consistently externalized in a way that weakens domain meaning. Prefer `Not Enough Evidence` when behavior ownership cannot be determined from the reviewed material.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant domain behavior and the available evidence supports that the domain model preserves meaningful decisions, rules, and state transitions within the model elements that own that behavior.

# Fail Conditions

Use `Fail` when direct evidence shows that meaningful domain behavior belonging to the domain model is consistently externalized into callers, application services, procedural coordinators, or technical structures, leaving domain model elements as passive data structures despite demonstrated behavioral responsibility within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, duplicated, emerging, or ambiguous preservation of domain behavior, but the available material does not justify a definitive `Fail`. Use `Warning` when some meaningful behavior is present in the model while related decisions or transitions appear scattered or externalized without enough evidence for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable domain models, meaningful domain behavior, domain decisions, state transitions, or behavior ownership concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify domain model elements, inspect meaningful domain behavior, determine whether behavior belongs to the model, distinguish passive domain models from legitimate data structures, inspect externalized behavior, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when loss of behavioral richness affects central domain concepts, broad domain behavior, critical decisions, repeated state transitions, many callers, or stable domain boundaries. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to behavior with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the domain model element, the meaningful behavior or decision, the model ownership of that behavior, and whether the behavior is preserved or externalized. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-001, DDD-006, DDD-007, DDD-008, DDD-012. These are conceptual dependencies for shared value object, entity, domain service, application service, domain decision, invariant, and behavior vocabulary, not procedural prerequisites for evaluating this rule.
