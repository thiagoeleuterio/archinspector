# Rule ID

DDD-004

# Title

Aggregates must enforce consistency boundaries

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure aggregates protect the consistency rules that belong within their transactional boundary.

# Applicability

Apply this rule when the reviewed material includes identifiable aggregates, aggregate-like domain structures, transactional consistency rules, domain invariants spanning related objects, or state transitions that must remain consistent within a domain boundary. The rule is relevant only when the reviewed scope provides enough evidence to inspect the aggregate boundary and the consistency rules that belong inside it.

# Rule Statement

Aggregates must enforce the consistency rules that belong within their transactional consistency boundary.

# Rationale

Aggregates define a boundary within which related domain state must remain consistent. When an aggregate does not enforce the rules that belong inside that boundary, invalid combinations of state can appear, callers may need to coordinate internal consistency themselves, and the domain model becomes harder to reason about under change or concurrent modification. When rules that do not belong inside the aggregate are forced into it, the aggregate boundary can also become too broad and obscure model ownership.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed aggregate or aggregate-like structure, its relevant internal members or related objects, the consistency rules or invariants associated with its boundary, and the behavior or state transitions that enforce or bypass those rules. Direct evidence may include type definitions, member definitions, construction paths, method behavior, state transition behavior, domain rule checks, transaction-related behavior, dependency relationships involving aggregate members, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish aggregate consistency boundaries from aggregate root access control, general entity lifecycle, value object invariant protection, application workflow validation, persistence transactions, database constraints, module boundaries, ports, adapters, gateways, repositories, and framework mechanisms. Naming conventions, aggregate labels, folder names, or physical grouping may support interpretation, but they must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the structure being evaluated is an aggregate or aggregate-like domain consistency boundary. Then inspect whether the aggregate protects the consistency rules that belong within that boundary through its domain behavior and controlled state transitions, and whether callers can create or leave internally inconsistent aggregate state within the reviewed scope.

Evaluate only aggregate consistency boundary enforcement. Do not use this rule to evaluate whether the Aggregate Root is the controlled access point for modifying aggregate state, which belongs to `DDD-005`. Do not use this rule to evaluate value object invariant protection, bounded context boundaries, domain language consistency, repository abstraction, application service coordination, persistence architecture, database transaction strategy, source dependency direction, ports and adapters, or Clean Architecture use case isolation unless that evidence directly shows whether the aggregate protects its own consistency boundary.

Interpret evidence conservatively. An object graph, collection, or transaction is not automatically an aggregate. A large object is not automatically a failed aggregate, and a small object is not automatically a passing aggregate. A finding requires evidence that a consistency rule belongs inside the aggregate boundary and is either protected, bypassed, inconsistently enforced, or externalized to callers that should not own that aggregate consistency.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant aggregates and their internal consistency rules, and the available evidence supports that those aggregates enforce the consistency rules belonging within their transactional boundary through domain behavior and controlled state transitions.

# Fail Conditions

Use `Fail` when direct evidence shows that an aggregate can enter internally inconsistent state, that callers are required to coordinate consistency rules that belong inside the aggregate boundary, that aggregate state transitions bypass required consistency checks, or that unrelated consistency concerns are forced into an aggregate boundary in a way that demonstrably blurs rule ownership within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, duplicated, indirect, or ambiguous enforcement of aggregate consistency boundaries, but the available material does not justify a definitive `Fail`. Use `Warning` when some state transitions appear protected while others are unclear, or when consistency ownership appears split between the aggregate and external callers without enough evidence for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable aggregates, aggregate-like structures, transactional consistency rules, related domain state, or aggregate boundary concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify aggregates, inspect aggregate boundaries, identify the consistency rules that belong inside the boundary, inspect state transitions, determine whether callers can bypass aggregate consistency, distinguish aggregate consistency from persistence transactions or application validation, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when weak aggregate consistency enforcement affects central aggregates, critical domain invariants, broad transaction behavior, many callers, or state transitions with significant business impact. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to an aggregate with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the aggregate boundary, the consistency rules belonging inside it, and whether the aggregate enforces or fails to enforce those rules. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002, DDD-003, DDD-012. These are conceptual dependencies for shared domain language, bounded context, aggregate, and invariant vocabulary, not procedural prerequisites for evaluating this rule.
