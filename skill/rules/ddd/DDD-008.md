# Rule ID

DDD-008

# Title

Application services must not contain domain decisions

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure application services coordinate domain behavior without owning domain rules or decisions.

# Applicability

Apply this rule when the reviewed material includes identifiable application services, application-level coordinators, command or request handlers, use-case-like application operations, or orchestration components that invoke domain model behavior. The rule is relevant only when the reviewed scope provides enough evidence to distinguish application coordination from domain decisions.

# Rule Statement

Application services must coordinate domain behavior without owning domain rules or domain decisions.

# Rationale

Application services coordinate work needed to fulfill an application operation, such as invoking domain behavior and arranging interactions around it. When application services contain domain decisions, domain rules become separated from the model elements that give them meaning, behavior may be duplicated across application flows, and the domain model becomes harder to reason about independently from application orchestration.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed application service or application-level coordinator, the domain model elements it invokes or coordinates, and the decisions or rules present in its behavior. Direct evidence may include type definitions, method signatures, constructor dependencies, type dependencies, method behavior, branching or decision logic, domain rule checks, state transition calls, invocation paths, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish application services from domain services, entities, value objects, aggregates, aggregate roots, repositories, factories, controllers, presenters, gateways, adapters, framework handlers, transaction mechanisms, and purely technical orchestration. Naming conventions, service suffixes, handler labels, folder names, or use-case labels may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the component being evaluated is an application service or application-level coordinator in the DDD sense. Then inspect whether its behavior coordinates domain model interactions or whether it owns domain decisions that should be expressed by domain model elements such as entities, value objects, aggregates, aggregate roots, or domain services.

Evaluate only application service ownership of domain decisions. Do not use this rule to evaluate the Clean Architecture Dependency Rule, Clean Architecture use case isolation, controller delegation, gateway isolation, source dependency direction, ports and adapters, framework leakage, persistence leakage, transaction management, logging, authorization, retries, or other technical orchestration unless that evidence directly shows application services owning domain decisions.

Interpret evidence conservatively. Branching, validation, transaction boundaries, or coordination in an application service is not automatically a violation unless evidence shows that the behavior represents a domain rule or domain decision rather than application flow or technical coordination. Prefer `Not Enough Evidence` when the reviewed material does not allow the reviewer to distinguish domain decisions from application coordination.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant application services and domain behavior, and the available evidence supports that application services coordinate domain model interactions without owning domain rules or domain decisions.

# Fail Conditions

Use `Fail` when direct evidence shows that an application service contains domain decisions, enforces domain rules that belong to domain model elements, determines domain outcomes that should be decided by the domain model, or duplicates domain decision logic across application operations within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, duplicated, indirect, or ambiguous domain decisions in an application service, but the available material does not justify a definitive `Fail`. Use `Warning` when an application service mostly coordinates domain behavior but contains limited decision logic whose domain ownership is unclear.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable application services, application-level coordinators, domain model interactions, or domain decision ownership concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify application services, distinguish application services from domain services or adapters, inspect behavior, identify domain model interactions, determine whether a decision is a domain decision, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when domain decisions in application services affect central domain behavior, broad application flows, critical business rules, multiple services, or stable domain boundaries. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to a minor decision with limited architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the application service role, the domain decision involved, the relevant domain model ownership, and whether the service owns or merely coordinates the decision. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002, DDD-006, DDD-007, DDD-013. These are conceptual dependencies for shared domain language, entity, domain service, application service, domain decision, and domain behavior vocabulary, not procedural prerequisites for evaluating this rule.
