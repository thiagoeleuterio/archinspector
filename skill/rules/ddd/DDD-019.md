# Rule ID

DDD-019

# Title

Domain code must remain isolated from non-domain responsibilities

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure domain code preserves domain meaning by avoiding responsibilities that belong to application coordination, technical mechanisms, or presentation concerns.

# Applicability

Apply this rule when the reviewed material includes identifiable domain code, domain model elements, domain behavior, domain rules, or domain services together with responsibilities that may belong to application coordination, presentation, delivery mechanisms, persistence mechanisms, messaging mechanisms, framework mechanisms, external integration mechanisms, or other non-domain concerns. The rule is relevant only when the reviewed scope provides enough evidence to distinguish domain responsibility from non-domain responsibility.

# Rule Statement

Domain code must remain focused on domain meaning, domain rules, and domain behavior without assuming application coordination, presentation, or technical mechanism responsibilities.

# Rationale

Domain code preserves the meaning of the model by expressing concepts, rules, decisions, and behavior that belong to the domain. When domain code assumes responsibilities for application workflow, presentation formatting, persistence mechanics, transport mechanics, framework lifecycle, messaging infrastructure, or external integration details, domain meaning becomes mixed with concerns that change for different reasons. This weakens the ability to reason about the domain model independently and can obscure which code owns domain decisions.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed domain code and the non-domain responsibilities that may be present in it. Direct evidence may include type definitions, member definitions, method behavior, imports, package references, type dependencies, constructor dependencies, domain rule checks, application workflow behavior, presentation formatting behavior, persistence behavior, transport behavior, messaging behavior, framework lifecycle usage, external integration behavior, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish domain responsibilities from application services, use case coordination, controllers, presenters, gateways, adapters, repositories as domain-facing abstractions, persistence implementations, framework configuration, messaging infrastructure, transport contracts, serialization concerns, and presentation models. Naming conventions, folder names, domain labels, service suffixes, or technical suffixes may support interpretation, but they must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the code being evaluated is domain code or a domain model element. Then inspect whether its behavior is focused on domain meaning, rules, and behavior, or whether it assumes responsibilities that belong to application coordination, presentation, delivery, persistence, messaging, framework, external integration, or other technical mechanisms.

Evaluate only non-domain responsibilities inside domain code. Do not use this rule to duplicate Hexagonal rules for dependency direction, framework concerns, persistence concerns, messaging concerns, adapter boundaries, or core testability. Do not use this rule to duplicate Clean Architecture rules for source dependency direction, use case isolation, controllers, presenters, gateways, data mappers, boundary data structures, or flow of control. Do not duplicate `DDD-008`, which evaluates domain decisions inside application services; `DDD-009`, which evaluates repositories as domain collection boundaries; or `DDD-013`, which evaluates whether domain models preserve behavioral richness. Use those concerns only when they directly show domain code assuming non-domain responsibility.

Interpret evidence conservatively. A technical dependency near domain code is not automatically a violation unless it shows that domain code assumes a non-domain responsibility. A domain model may expose domain-facing abstractions such as repository concepts when the reviewed material supports that role. A finding requires evidence that non-domain coordination, presentation, or technical mechanism responsibility is located in domain code and affects domain meaning, behavior, or boundary integrity.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant domain code and potential non-domain responsibilities, and the available evidence supports that domain code remains focused on domain meaning, rules, and behavior without assuming application coordination, presentation, or technical mechanism responsibilities.

# Fail Conditions

Use `Fail` when direct evidence shows that domain code performs application workflow coordination, presentation formatting, controller or presenter responsibilities, persistence implementation mechanics, transport or serialization mechanics, messaging infrastructure behavior, framework lifecycle or configuration work, external integration calls, or other non-domain responsibilities within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, emerging, or ambiguous non-domain responsibility inside domain code, but the available material does not justify a definitive `Fail`. Use `Warning` when domain code mostly preserves domain focus but includes limited behavior that appears technical, presentational, or coordinative without enough evidence for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable domain code, domain model elements, domain behavior, domain rules, or non-domain responsibility concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify domain code, inspect behavior or dependencies, distinguish domain responsibility from application coordination or technical mechanisms, determine whether non-domain responsibilities are assumed by domain code, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when non-domain responsibilities in domain code affect central domain models, critical domain rules, broad domain behavior, many callers, stable domain boundaries, or the ability to reason about domain meaning independently. Assign lower severity when the concern is narrow, isolated, peripheral, indirect, or limited to a small domain area with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the domain code role, the non-domain responsibility involved, and whether domain code assumes that responsibility. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming or folder location alone must not produce `Confirmed` confidence.

# Dependencies

DDD-007, DDD-008, DDD-009, DDD-012, DDD-013. These are conceptual dependencies for shared domain service, application service, repository, invariant, domain behavior, and domain responsibility vocabulary, not procedural prerequisites for evaluating this rule.
