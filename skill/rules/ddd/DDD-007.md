# Rule ID

DDD-007

# Title

Domain services must contain domain behavior only when it does not belong to an entity or value object

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure domain services express domain operations that do not naturally belong inside entities or value objects.

# Applicability

Apply this rule when the reviewed material includes identifiable domain services, service-like domain model elements, domain operations, behavior involving multiple domain objects, or domain decisions whose natural owner may be a domain service, entity, or value object. The rule is relevant only when the reviewed scope provides enough evidence to inspect the domain behavior and determine whether it belongs in a domain service or in another domain model element.

# Rule Statement

Domain services must contain domain behavior only when that behavior does not naturally belong to an entity or value object.

# Rationale

Domain services express domain operations that are meaningful in the domain model but do not naturally fit inside a single entity or value object. When behavior that belongs to an entity or value object is moved into a domain service, the model can become procedural and the object that owns the domain meaning becomes passive. When a domain service contains application coordination or technical work instead of domain behavior, the service no longer represents a domain modeling concept.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed domain service or service-like domain element, the domain behavior it contains, and the entities, value objects, aggregates, or other domain concepts related to that behavior. Direct evidence may include type definitions, method signatures, method behavior, domain rule checks, state transition behavior, dependencies on domain model elements, construction or invocation paths, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish domain services from application services, infrastructure services, adapters, gateways, repositories, factories, controllers, presenters, data mappers, utility services, framework services, and transaction or workflow coordinators. Naming conventions, service suffixes, folder names, or domain labels may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the service being evaluated is a domain service or service-like domain model element. Then inspect whether the behavior it contains is domain behavior and whether that behavior naturally belongs outside a single entity or value object, such as when the operation spans multiple domain concepts or expresses a domain operation without an obvious object owner.

Evaluate only domain service responsibility. Do not use this rule to evaluate application services, use case orchestration, technical coordination, transaction management, dependency direction, ports and adapters, Clean Architecture use case isolation, controller delegation, gateway isolation, repository abstraction, factory creation, aggregate consistency boundaries, or value object invariant protection unless that evidence directly shows whether a domain service contains behavior that belongs in an entity or value object.

Interpret evidence conservatively. A service name is not enough to prove domain service intent, and the mere presence of domain objects in a service is not enough to prove a violation. A finding requires evidence that the behavior is a domain operation and that its natural ownership is either correctly outside an entity or value object or incorrectly displaced from one. Prefer `Not Enough Evidence` when behavior ownership cannot be determined from the reviewed material.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant domain services and the domain behavior they contain, and the available evidence supports that the behavior is domain behavior that does not naturally belong to an entity or value object within the reviewed domain model.

# Fail Conditions

Use `Fail` when direct evidence shows that a domain service contains behavior that naturally belongs to an entity or value object, turns entity or value object behavior into externally coordinated procedure, or contains application coordination or technical responsibilities instead of domain behavior within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, duplicated, indirect, or ambiguous domain behavior placement in a domain service, but the available material does not justify a definitive `Fail`. Use `Warning` when a domain service mostly contains appropriate domain operations but some behavior appears to belong to an entity or value object without enough evidence for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable domain services, service-like domain elements, domain operations, entities or value objects related to the behavior, or domain service responsibility concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify domain services, distinguish domain services from application or technical services, inspect service behavior, identify related entities or value objects, determine natural behavior ownership, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when misplaced domain service behavior affects central domain operations, broad domain behavior, critical rules, multiple entities or value objects, or stable domain boundaries. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to a service operation with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the domain service role, the domain behavior involved, the relevant entity or value object ownership question, and whether behavior placement is appropriate. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-001, DDD-002, DDD-006, DDD-013. These are conceptual dependencies for shared value object, domain language, entity, domain service, and domain behavior vocabulary, not procedural prerequisites for evaluating this rule.
