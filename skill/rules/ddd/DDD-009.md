# Rule ID

DDD-009

# Title

Repositories must represent domain collection boundaries

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure repositories expose domain-oriented collection access without making the domain model conform to storage concerns.

# Applicability

Apply this rule when the reviewed material includes identifiable repositories, repository-like domain abstractions, aggregate or entity collection access, domain retrieval or storage intent, or domain-facing contracts used to obtain and persist domain model elements. The rule is relevant only when the reviewed scope provides enough evidence to inspect whether the repository represents a domain collection boundary.

# Rule Statement

Repositories must represent domain-oriented collection boundaries without making the domain model conform to storage concerns.

# Rationale

Repositories provide a domain-facing collection abstraction for accessing aggregates or entities according to domain meaning. When repositories expose storage concerns instead of domain collection concepts, the domain model can become shaped by persistence structures, query mechanisms, or technical data access details. When repository responsibilities are unclear, callers may reason about storage mechanics instead of domain collection boundaries.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed repository or repository-like abstraction, the domain model elements it collects or retrieves, the domain-facing operations it exposes, and any storage concerns that shape those operations or the domain model. Direct evidence may include type definitions, method signatures, return and parameter types, type dependencies, constructor dependencies, method behavior, query or collection operation names tied to domain meaning, aggregate or entity references, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish repositories as domain collection abstractions from persistence adapters, ORM mechanisms, database contexts, gateways, outbound ports, data mappers, infrastructure services, query handlers, technical data access objects, serialization structures, and framework-specific storage APIs. Naming conventions, repository suffixes, folder names, or storage-related labels may support interpretation but must not be treated as conclusive evidence without corroborating structural, behavioral, or contract evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the component being evaluated is a repository or repository-like domain collection abstraction. Then inspect whether its contract expresses domain collection access for aggregates or entities and whether storage concerns shape the repository contract or make the domain model conform to persistence needs.

Evaluate only repositories as Domain-Driven Design domain collection boundaries. Do not use this rule to evaluate persistence implementation, ORM configuration, database schema design, infrastructure adapters, Hexagonal outbound ports, Clean Architecture gateways, data mapper behavior, general dependency direction, query optimization, transaction management, or framework isolation unless that evidence directly shows whether a repository boundary expresses domain collection access or storage-shaped domain modeling.

Interpret evidence conservatively. A type named repository is not automatically a DDD repository, and storage terminology is not automatically a violation unless evidence shows that storage concerns shape the domain-facing collection boundary or domain model. A finding requires evidence of the repository role, the domain collection it represents, and the relevant storage-shaped or domain-oriented contract behavior.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant repositories, collected domain model elements, and repository contracts, and the available evidence supports that repositories expose domain-oriented collection access without making the domain model conform to storage concerns.

# Fail Conditions

Use `Fail` when direct evidence shows that a repository exposes storage mechanisms, persistence models, schema-shaped operations, technical query APIs, infrastructure data structures, or storage lifecycle concerns as domain-facing collection boundaries, or forces aggregates or entities to conform to storage concerns within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, or ambiguous repository collection boundaries, but the available material does not justify a definitive `Fail`. Use `Warning` when a repository mostly expresses domain collection access but some contract members appear storage-shaped without enough evidence for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable repositories, repository-like domain abstractions, aggregate or entity collection access, domain-facing retrieval or persistence contracts, or repository responsibility concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify repositories, distinguish repositories from persistence adapters or gateways, inspect repository contracts, identify collected aggregates or entities, determine whether operations are domain-oriented or storage-shaped, determine whether the domain model conforms to storage concerns, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when storage-shaped repository boundaries affect central aggregates or entities, broad domain collection access, critical domain operations, many callers, or stable domain boundaries. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to a repository with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the repository role, the collected domain model elements, the repository contract shape, and whether it represents a domain collection boundary or storage-shaped boundary. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002, DDD-003, DDD-004, DDD-005, DDD-006. These are conceptual dependencies for shared domain language, bounded context, aggregate, Aggregate Root, entity, and repository vocabulary, not procedural prerequisites for evaluating this rule.
