# Rule ID

DDD-010

# Title

Factories must protect complex domain creation

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure factories preserve valid construction when domain object creation involves non-trivial invariants or lifecycle rules.

# Applicability

Apply this rule when the reviewed material includes identifiable factories, factory-like domain creation behavior, complex construction paths, lifecycle creation decisions, non-trivial object composition, or creation-time invariants for domain model elements. The rule is relevant only when the reviewed scope provides enough evidence to inspect whether domain object creation requires protection beyond simple direct construction.

# Rule Statement

Factories must protect domain object creation when construction involves non-trivial invariants, lifecycle decisions, or domain composition that must produce a valid domain model state.

# Rationale

Factories protect domain meaning when creating a valid object requires more than assigning simple values. When complex creation rules are scattered across callers, callers can bypass required invariants, duplicate lifecycle decisions, or assemble domain objects in inconsistent combinations. When a factory is required by the domain model, its responsibility is to make valid construction explicit without turning persistence, dependency injection, framework wiring, or technical object construction into a domain modeling concern.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed factory or factory-like creation behavior, the domain model element being created, and the creation-time invariants, lifecycle decisions, or composition rules associated with valid construction. Direct evidence may include type definitions, factory method signatures, construction paths, constructor behavior, method behavior, state initialization behavior, domain rule checks, dependency relationships between created domain objects, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish domain factories from dependency injection factories, persistence materialization mechanisms, framework object construction, technical builders, test data builders, serialization constructors, data mappers, repositories, application services, and infrastructure composition. Naming conventions, factory suffixes, folder names, or creation labels may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the creation behavior being evaluated is domain creation rather than technical object construction. Then inspect whether creation involves non-trivial domain invariants, lifecycle decisions, or composition rules that must be protected during construction, and whether the factory or equivalent creation boundary prevents invalid domain object creation within the reviewed scope.

Evaluate only factory responsibility for complex domain creation. Do not require a factory for every entity, value object, aggregate, aggregate root, or domain object. Do not use this rule to evaluate persistence materialization, dependency injection, object lifetime management, framework construction, repository collection boundaries, aggregate consistency enforcement after creation, Aggregate Root access control, application service orchestration, source dependency direction, ports and adapters, or Clean Architecture boundary rules unless that evidence directly shows whether complex domain creation is protected.

Interpret evidence conservatively. A type named factory is not automatically a domain factory, and direct constructors are not automatically a violation. A finding requires evidence that creation complexity belongs to the domain model and that the reviewed creation path protects, bypasses, duplicates, or inconsistently enforces the required creation rules. Prefer `Not Enough Evidence` when the reviewed material does not allow the reviewer to distinguish simple construction from protected domain creation.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant complex domain creation paths and the available evidence supports that factories or equivalent domain creation boundaries protect required creation-time invariants, lifecycle decisions, and non-trivial composition rules.

# Fail Conditions

Use `Fail` when direct evidence shows that domain object creation involving non-trivial invariants, lifecycle decisions, or required composition can be bypassed, duplicated across callers, inconsistently enforced, or performed through construction paths that allow invalid domain model state within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, duplicated, emerging, or ambiguous protection of complex domain creation, but the available material does not justify a definitive `Fail`. Use `Warning` when some creation paths appear protected while others are unclear, or when a factory exists but its domain creation responsibility is only partially supported by evidence.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable factories, factory-like domain creation behavior, complex domain construction, creation-time invariants, lifecycle creation decisions, or non-trivial domain composition relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify domain creation paths, distinguish domain factories from technical construction mechanisms, inspect creation behavior, identify creation-time invariants or lifecycle decisions, determine whether construction can produce invalid domain state, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when weak protection of complex domain creation affects central aggregates, entities, value objects, lifecycle entry points, critical invariants, broad creation surfaces, or many callers. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to creation behavior with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the domain creation responsibility, the relevant creation-time invariants or lifecycle decisions, and whether the creation path protects or fails to protect valid construction. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-001, DDD-004, DDD-006, DDD-012. These are conceptual dependencies for shared value object, aggregate, entity, lifecycle, creation, and invariant vocabulary, not procedural prerequisites for evaluating this rule.
