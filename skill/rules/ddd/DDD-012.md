# Rule ID

DDD-012

# Title

Domain invariants must be enforced by the domain model

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure domain rules that must always hold are protected by domain behavior and state transitions.

# Applicability

Apply this rule when the reviewed material includes identifiable domain invariants, domain rules that must always remain true, domain model state transitions, creation behavior, or mutation behavior where invalid domain state must be prevented. The rule is relevant only when the reviewed scope provides enough evidence to inspect how mandatory domain invariants are protected during creation and state change.

# Rule Statement

Domain invariants that must always hold must be enforced by the domain model during creation and state transitions.

# Rationale

Mandatory domain invariants define conditions that must remain true for the model to represent valid domain state. When those invariants depend only on callers, application coordination, external validation, persistence constraints, or technical mechanisms, invalid state can enter the domain model and domain rule ownership becomes harder to reason about. The domain model should protect invariants that belong to it without turning this rule into a duplicate of value object, aggregate boundary, or Aggregate Root access evaluations.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed domain invariant, the domain model element or model boundary responsible for it, and the creation or state transition behavior that enforces or bypasses it. Direct evidence may include type definitions, member definitions, construction paths, method behavior, state transition behavior, domain rule checks, invariants described in documentation tied to reviewed domain code, dependency relationships involving domain model elements, reviewed code excerpts, or tests when they directly show domain invariant behavior.

Evidence must distinguish domain invariants from application input validation, user interface validation, persistence constraints, database rules, framework validation attributes, serialization requirements, transport validation, authorization checks, workflow preconditions, and technical consistency checks. Naming conventions, invariant labels, validation method names, or domain folder names may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the rule being evaluated is a mandatory domain invariant rather than an external validation or technical constraint. Then inspect whether the domain model protects that invariant during creation and state transitions, and whether callers can leave the model in invalid domain state by bypassing domain behavior.

Evaluate only the transversal responsibility for mandatory domain invariant enforcement by the domain model. Do not use this rule to duplicate value object invariant protection in `DDD-001`, aggregate transactional consistency boundaries in `DDD-004`, or Aggregate Root access control in `DDD-005`. Do not use this rule to evaluate factory responsibility for complex creation in `DDD-010`, application service ownership of domain decisions in `DDD-008`, persistence architecture, database constraint design, framework validation, source dependency direction, ports and adapters, or Clean Architecture boundary rules unless that evidence directly shows whether mandatory domain invariants depend only on external coordination.

Interpret evidence conservatively. The presence of validation outside the domain model is not automatically a violation when the checked condition is application input validation, boundary validation, or a rule that does not need to remain true inside the domain model. A finding requires evidence that a domain invariant must always hold and that the model protects, fails to protect, inconsistently protects, or externalizes that invariant within the reviewed scope.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant mandatory domain invariants and the available evidence supports that the domain model enforces those invariants during creation and state transitions without relying only on external coordination.

# Fail Conditions

Use `Fail` when direct evidence shows that a mandatory domain invariant can be violated through domain model creation or state transition paths, that callers must enforce an invariant that belongs to the domain model, or that enforcement exists only in application, presentation, persistence, framework, or other external mechanisms within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, duplicated, emerging, or ambiguous enforcement of mandatory domain invariants, but the available material does not justify a definitive `Fail`. Use `Warning` when some transitions protect an invariant while others are unclear, or when invariant ownership appears split between the domain model and external coordination without enough evidence for a confirmed violation.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable mandatory domain invariants, domain model state, creation behavior, state transitions, mutation behavior, or domain invariant enforcement concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify mandatory domain invariants, distinguish domain invariants from external validation or technical constraints, inspect creation or state transition behavior, determine whether invalid domain state can occur, identify invariant ownership, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when weak domain invariant enforcement affects central domain rules, critical valid-state guarantees, broad model behavior, many callers, aggregate or entity lifecycles, or stable domain boundaries. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to an invariant with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the mandatory domain invariant, the domain model responsibility for protecting it, the relevant creation or state transition paths, and whether enforcement is present or bypassed. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-001, DDD-004, DDD-005, DDD-006, DDD-010. These are conceptual dependencies for shared value object, aggregate, Aggregate Root, entity, creation, state transition, and invariant vocabulary, not procedural prerequisites for evaluating this rule.
