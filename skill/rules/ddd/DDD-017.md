# Rule ID

DDD-017

# Title

Published languages must define stable inter-context contracts

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure inter-context communication uses stable language owned by the publishing context.

# Applicability

Apply this rule when the reviewed material includes inter-context communication, domain-facing contracts, integration contracts, events, messages, APIs, files, documents, or other contract surfaces that appear to communicate domain meaning from one bounded context to another. The rule is relevant only when the reviewed scope provides enough evidence to inspect whether a stable Published Language exists and is governed by the publishing context.

# Rule Statement

Published languages must define stable, understandable inter-context contracts governed by the bounded context that publishes them.

# Rationale

A Published Language provides a stable language for other contexts to consume without requiring them to depend on the publisher's internal model. When inter-context contracts lack stable published meaning, consumers may infer domain semantics from implementation details, internal structures can become accidental contracts, and changes inside the publishing context can unexpectedly break downstream understanding.

# Evidence Required

Evaluation requires traceable evidence that identifies the publishing context, the consuming context or contexts, the inter-context contract, and the language or semantics exposed through that contract. Direct evidence may include contract definitions, schema or message definitions when tied to domain meaning, API contracts, event definitions, documentation, type definitions, versioned contract surfaces, dependency relationships, method signatures, reviewed code excerpts, or governance documentation tied to the publishing context.

Evidence must distinguish Published Language from internal domain model classes, generic DTOs, transport envelopes, serialization formats, protocol mechanisms, message brokers, persistence schemas, technical API routing, or naming conventions. Contract names, DTO suffixes, message labels, or version markers may support interpretation, but they must not be treated as conclusive evidence without corroborating domain-language, ownership, stability, contract, or documentation evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that one bounded context publishes a contract consumed by another context. Then inspect whether the contract exposes a stable language that is understandable to consumers, governed by the publishing context, and distinct from accidental exposure of internal implementation structures.

Evaluate only Published Language as a stable inter-context domain contract. Do not use this rule to evaluate transport mechanisms, serialization technology, broker configuration, technical versioning in isolation, generic API design, DTO naming, Hexagonal adapter model leakage, Clean Architecture boundary data structures, or domain event significance unless that evidence directly shows whether a Published Language defines stable inter-context meaning.

Interpret evidence conservatively. A DTO, event, message, or API is not a Published Language merely because of its name or because it crosses a boundary. Technical version numbers or schemas are not enough by themselves unless the reviewed material shows stable domain meaning and publisher governance. Prefer `Not Enough Evidence` when the reviewed material does not allow the reviewer to distinguish published domain language from transport or implementation detail.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify the publishing context, consuming context, and inter-context contract, and the available evidence supports that the contract uses a stable, understandable Published Language governed by the publishing context.

# Fail Conditions

Use `Fail` when direct evidence shows that inter-context consumers depend on internal model structures, unstable implementation details, ambiguous contract language, consumer-invented meanings, or contract surfaces not governed by the publishing context within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, emerging, or ambiguous Published Language stability, understandability, or governance, but the available material does not justify a definitive `Fail`. Use `Warning` when contracts appear published but their domain meaning, publisher ownership, or stability is only partially supported by evidence.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include inter-context communication, published contract surfaces, consuming contexts, publishing contexts, or Published Language concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the publishing context, identify consuming contexts, inspect inter-context contracts, determine contract language or semantics, distinguish Published Language from technical DTOs or transport structures, inspect governance, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when unstable or unclear Published Language affects central inter-context contracts, critical domain communication, many consumers, broad integration surfaces, or stable context boundaries. Assign lower severity when the issue is narrow, isolated, peripheral, consumer-limited, or confined to a small contract with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the publishing context, the consuming context, the inter-context contract, the published language, and whether its stability and governance are present. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002, DDD-003, DDD-014. These are conceptual dependencies for shared domain language, bounded context, context relationship, and inter-context contract vocabulary, not procedural prerequisites for evaluating this rule.
