# Rule ID

DDD-018

# Title

Upstream and downstream roles must be visible in context relationships

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure directional influence between bounded contexts can be identified from reviewed material.

# Applicability

Apply this rule when the reviewed material includes relationships between bounded contexts or candidate contexts where one context may influence, publish to, depend on, conform to, consume from, or otherwise affect another context. The rule is relevant only when the reviewed scope provides enough evidence to inspect directional influence between the participating contexts.

# Rule Statement

Upstream and downstream roles in bounded context relationships must be visible enough to identify the direction of influence between participating contexts.

# Rationale

Upstream and downstream roles make directional influence in a context relationship visible. When these roles are unclear, reviewers cannot reliably determine which context publishes language, which context conforms or translates, which side can change independently, or where negotiation and change impact should be evaluated. Visible roles reduce ambiguity around dependency, influence, and responsibility in context relationships.

# Evidence Required

Evaluation requires traceable evidence that identifies the participating contexts, the relationship between them, and the direction of influence or dependency. Direct evidence may include context relationship documentation, published contracts, consuming contracts, dependency relationships, invocation paths, translation behavior, ownership of contract language, change governance documentation, type dependencies, method behavior, reviewed code excerpts, or documentation tied to the involved domain models.

Evidence must distinguish upstream/downstream influence from source dependency direction inside a single system, call direction, deployment topology, package reference direction, team hierarchy, or generic client/server labels. Naming conventions, upstream or downstream labels, folder names, service names, or diagrams may support interpretation, but they must not be treated as conclusive evidence without corroborating relationship, contract, influence, dependency, or documentation evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that a bounded context relationship exists between participating contexts. Then inspect whether the reviewed material makes the upstream and downstream roles visible enough to identify which context influences the contract, language, or model relationship, and which context consumes, conforms to, translates, or is affected by that influence.

Evaluate only visibility of upstream and downstream roles and direction of influence in context relationships. Do not duplicate `DDD-014`, which evaluates whether the relationship itself is explicit enough in general. Do not use this rule to evaluate source dependency direction within a system, Hexagonal dependency direction toward the core, Clean Architecture dependency direction toward policies, transport call direction, messaging topology, or deployment relationships unless that evidence directly shows context-level upstream/downstream influence.

Interpret evidence conservatively. A caller is not automatically downstream, and a callee is not automatically upstream. A project dependency direction is not enough by itself to confirm context influence. Prefer `Not Enough Evidence` when a relationship appears to exist but the reviewed material cannot show which context influences the other.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant context relationships and the available evidence supports that upstream and downstream roles are visible enough to determine directional influence between the participating contexts.

# Fail Conditions

Use `Fail` when direct evidence shows that bounded contexts have a relationship with domain, contract, or model influence, but the upstream and downstream roles are hidden, contradictory, or unclear enough that the direction of influence cannot be identified within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, emerging, or ambiguous visibility of upstream and downstream roles, but the available material does not justify a definitive `Fail`. Use `Warning` when roles are visible for some interactions but unclear for others, or when directional influence can be inferred only weakly from multiple indirect signals.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include bounded context relationships, candidate context relationships, inter-context influence, published or consumed contracts, model conformance, or upstream/downstream role concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify participating contexts, establish a context relationship, inspect directional influence, distinguish upstream/downstream roles from call or source dependency direction, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when unclear upstream/downstream roles affect central context relationships, critical published language, broad model influence, many consumers, or the ability to reason about change impact across contexts. Assign lower severity when role ambiguity is narrow, isolated, peripheral, or limited to a small relationship with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the participating contexts, the relationship between them, the direction of influence, and whether upstream/downstream roles are visible. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming, labels, or call direction alone must not produce `Confirmed` confidence.

# Dependencies

DDD-003, DDD-014, DDD-017. These are conceptual dependencies for shared bounded context, context relationship, published language, and directional influence vocabulary, not procedural prerequisites for evaluating this rule.
