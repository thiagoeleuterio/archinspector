# Rule ID

DDD-011

# Title

Domain events must express domain-significant facts

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure domain events represent meaningful facts from the domain model rather than technical notifications.

# Applicability

Apply this rule when the reviewed material includes identifiable domain events, event-like domain model elements, event records emitted by domain behavior, or event names and payloads intended to represent something significant that occurred in the domain. The rule is relevant only when the reviewed scope provides enough evidence to inspect event meaning, event origin, and the domain fact represented by the event.

# Rule Statement

Domain events must express domain-significant facts that have already occurred in the domain model.

# Rationale

Domain events make important domain facts visible after they occur. When an event represents a technical notification, command, transport message, persistence change, or implementation signal instead of a domain-significant fact, reviewers and maintainers may mistake technical workflow for domain meaning. Domain events should preserve domain language and factual past occurrence without absorbing messaging infrastructure, delivery guarantees, or distributed consistency concerns.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed domain event or event-like structure, the domain behavior or state transition that produces it, and the domain fact it represents. Direct evidence may include type definitions, event member definitions, event names tied to domain language, method behavior that records or raises events, state transition behavior, domain rule checks associated with event creation, dependencies involving event producers or consumers, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish domain events from integration events, technical notifications, commands, messages, callbacks, persistence hooks, audit records, logging records, framework events, messaging contracts, outbox records, broker payloads, and transport envelopes. Naming conventions, event suffixes, folder names, or message labels may support interpretation but must not be treated as conclusive evidence without corroborating structural, behavioral, or domain-language evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the event being evaluated is intended to be a domain event rather than a technical or integration message. Then inspect whether the event expresses a domain-significant fact that has already occurred, whether its name and data preserve domain meaning, and whether the event is produced from domain behavior or state transition evidence rather than only from technical workflow.

Evaluate only the domain meaning of domain events. Do not use this rule to evaluate publication mechanisms, message transport, broker configuration, retries, delivery guarantees, outbox implementation, eventual consistency strategy, distributed transaction design, messaging adapter boundaries, source dependency direction, use case orchestration, or integration event contract stability unless that evidence directly shows whether an event represents a domain-significant fact.

Interpret evidence conservatively. A type named as an event is not automatically a domain event, and an event emitted by application or infrastructure code is not automatically invalid unless evidence shows that it is being treated as a domain event without representing a domain fact. A finding requires evidence of the event role, the relevant domain meaning, and whether the event expresses a past domain fact or merely a technical notification, command, or transport concern.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant domain events and the available evidence supports that they represent domain-significant facts that have already occurred, using domain language and data tied to the domain behavior or state transition that produced them.

# Fail Conditions

Use `Fail` when direct evidence shows that a domain event represents a technical notification, command intent, persistence signal, transport concern, framework callback, or implementation workflow instead of a domain-significant fact that has already occurred within the reviewed domain model.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, duplicated, emerging, or ambiguous domain significance in events, but the available material does not justify a definitive `Fail`. Use `Warning` when an event appears mostly domain-oriented but includes unclear technical meaning, ambiguous tense, weak connection to domain behavior, or payload shape whose domain significance is only partially supported.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable domain events, event-like domain model elements, domain behavior that emits events, domain state transitions represented by events, or domain event meaning concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify domain events, distinguish domain events from technical or integration messages, inspect event creation behavior, identify the domain fact represented, determine whether the event describes something already occurred, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when weak domain event meaning affects central domain processes, broad event-driven behavior, critical domain facts, many consumers, or stable domain boundaries. Assign lower severity when the issue is narrow, isolated, peripheral, or limited to an event with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the domain event role, the domain fact represented, the behavior or state transition that produced it, and whether it expresses a domain-significant fact. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-002, DDD-003, DDD-004, DDD-006. These are conceptual dependencies for shared domain language, bounded context, aggregate, entity, state transition, and domain fact vocabulary, not procedural prerequisites for evaluating this rule.
