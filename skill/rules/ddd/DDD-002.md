# Rule ID

DDD-002

# Title

Domain language must be reflected consistently

# Category

Domain-Driven Design

# Status

Active

# Intent

Ensure domain terminology is used consistently across the reviewed domain model and related boundaries.

# Applicability

Apply this rule when the reviewed material includes identifiable domain concepts, domain model structures, domain behavior, domain-facing contracts, or boundary interactions where domain terminology is used to express business meaning. The rule is relevant only when the reviewed scope provides enough evidence to compare the meaning of domain terms across the domain model and its immediate domain-facing boundaries.

# Rule Statement

Domain language must be reflected consistently across the domain model and its immediate domain-facing boundaries.

# Rationale

Consistent domain language preserves shared meaning between the domain model and the business concepts it represents. When the same term is used for different meanings, or different terms are used for the same domain concept without a visible distinction, reviewers and maintainers have weaker evidence for domain behavior, model boundaries, and rule ownership. Inconsistent language can also hide boundary confusion by making incompatible meanings appear interchangeable.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed domain terminology, the domain concepts represented by that terminology, and the domain model or immediate domain-facing boundaries where those terms appear. Direct evidence may include type definitions, member definitions, method behavior, domain-facing contracts, boundary data structures, domain rule checks, state transition behavior, dependency relationships involving domain concepts, reviewed code excerpts, or documentation tied to the reviewed domain model.

Evidence must distinguish domain language from isolated class names, folder names, namespace labels, technical naming conventions, transport labels, persistence labels, serialization labels, framework labels, and presentation labels. Naming conventions may support interpretation, but naming alone must not be treated as conclusive evidence of consistent or inconsistent domain language without corroborating structural, behavioral, boundary, or documentation evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible conclusion that the language being evaluated represents domain terminology rather than technical naming or implementation labels. Then inspect whether the same domain concept is expressed consistently across domain behavior, model structures, state transitions, and immediate domain-facing boundaries, and whether distinct domain concepts remain distinguishable when similar terminology is used.

Evaluate only consistency of domain language in the domain model and its immediate domain-facing boundaries. Do not use this rule to evaluate class naming conventions, folder aesthetics, namespace organization, code style, bounded context integrity as a whole, aggregate consistency boundaries, aggregate root access control, ports and adapters, Clean Architecture use case boundaries, dependency direction, framework isolation, persistence design, or transport model design unless that evidence directly shows whether domain language preserves or blurs domain meaning.

Interpret evidence conservatively. A term appearing in multiple places is not automatically consistent, and a synonym is not automatically inconsistent. A finding requires evidence that terminology preserves domain meaning, creates ambiguity, conflates distinct meanings, fragments a single meaning, or conflicts with documented or behavioral domain usage within the reviewed scope. Prefer `Not Enough Evidence` when the reviewed material does not allow the reviewer to connect terms to domain meaning.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to identify relevant domain terminology and the available evidence supports that domain terms are used consistently across the domain model and immediate domain-facing boundaries, with distinct meanings kept distinguishable and shared meanings expressed without unsupported ambiguity.

# Fail Conditions

Use `Fail` when direct evidence shows that domain terminology is used inconsistently in a way that blurs domain meaning, conflates distinct business concepts, splits the same domain concept into conflicting terms without a visible distinction, or causes domain behavior, state transitions, or boundary interactions to rely on incompatible meanings within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, duplicated, emerging, or ambiguous domain language, but the available material does not justify a definitive `Fail`. Use `Warning` when terminology appears mostly consistent but some domain-facing boundaries, behaviors, or model structures use terms whose meaning is unclear or only partially aligned.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include identifiable domain terminology, domain model structures, domain behavior, domain-facing boundaries, or business concepts relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify domain terminology, connect terms to domain meaning, compare terminology across the domain model and immediate boundaries, distinguish domain language from technical naming, determine whether similar terms represent the same or different concepts, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when inconsistent domain language affects central domain concepts, broad domain behavior, bounded context interpretation, critical business rules, or many domain-facing interactions. Assign lower severity when the inconsistency is narrow, isolated, peripheral, or limited to terminology with minor architectural impact. Severity must reflect the demonstrated domain and maintainability impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the domain terminology, the related domain meaning, and whether language is used consistently or inconsistently across the reviewed domain model and immediate boundaries. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

DDD-003. This is a conceptual dependency for shared bounded context and domain model boundary vocabulary, not a procedural prerequisite for evaluating this rule.
