# Rule ID

HEX-007

# Title

Dependencies must point toward the core

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether dependencies between adapters and the application core point toward the core and do not make the core depend on adapters within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes an application core and one or more inbound adapters, outbound adapters, delivery mechanisms, infrastructure components, or external integration components. The rule is relevant only when dependency direction between the core and those external-facing components can be inspected.

# Rule Statement

Dependencies must point toward the core.

# Rationale

Dependency direction toward the core protects application and domain behavior from external mechanisms, keeps adapters replaceable, and prevents delivery or infrastructure decisions from becoming required dependencies of core behavior.

# Evidence Required

Evaluation requires traceable evidence that identifies the application core scope, the reviewed adapter or external-facing scopes, and the dependency relationships between them. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, or reviewed code excerpts showing whether adapters depend on core ports or behavior and whether core code depends on adapter implementations or adapter-specific concerns. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between the application core and adapters or external-facing components. Then evaluate the direction of dependencies using the strongest available evidence, giving priority to project or module references before weaker local signals. Interpret evidence conservatively, and do not infer dependency direction from intended architecture, naming, or missing material. This rule evaluates the general dependency direction between core and adapters; more specialized leakage through framework, persistence, messaging, or adapter model concerns should be evaluated by the corresponding specialized rules when applicable.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant core-adapter dependencies and the available evidence supports that adapters depend on the core while the core does not depend on adapters.

# Fail Conditions

Use `Fail` when direct evidence shows that application core code depends on inbound adapter implementations, outbound adapter implementations, delivery mechanisms, infrastructure components, or other adapter-specific concerns within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates a partial, indirect, inconsistent, cyclic, or ambiguous dependency from the core toward adapters, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include an application core, adapters, delivery mechanisms, infrastructure components, external integration components, or dependency relationships relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the application core scope, identify adapter scopes, inspect dependency direction, distinguish adapter dependencies from core dependencies, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when reversed or cyclic dependencies affect central core behavior, multiple adapters, broad integration surfaces, or stable architectural boundaries. Assign lower severity when the dependency is narrow, isolated, indirect, or limited to peripheral adapter interactions. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant scopes and dependency direction. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

HEX-001, HEX-002, HEX-004, HEX-005. These are conceptual dependencies for shared core, port, and adapter vocabulary, not procedural prerequisites for evaluating this rule.
