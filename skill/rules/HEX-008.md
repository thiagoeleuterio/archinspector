# Rule ID

HEX-008

# Title

Framework concerns must remain outside the core

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether framework-specific APIs and configuration remain outside application core behavior within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes application core behavior and one or more framework-specific APIs, framework configuration mechanisms, runtime integration mechanisms, or framework-driven components that may interact with the core. The rule is relevant only when there is enough context to distinguish core behavior from framework concerns.

# Rule Statement

Framework concerns must remain outside the core.

# Rationale

Framework-specific APIs and configuration in the application core couple core behavior to technical runtime mechanisms, reduce portability across delivery or infrastructure choices, and make application behavior harder to reason about independently from framework lifecycle and configuration concerns.

# Evidence Required

Evaluation requires traceable evidence that identifies the application core scope and the reviewed framework concerns. Direct evidence may include project or module references, package references, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, configuration declarations, attributes or annotations, or reviewed code excerpts showing whether core behavior depends on framework APIs, framework lifecycle hooks, framework configuration types, or framework-specific integration contracts. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between application core behavior and framework concerns. Then evaluate whether framework-specific APIs or configuration appear inside core behavior or remain isolated in adapters, composition boundaries, or other external-facing infrastructure. Interpret evidence conservatively, and do not treat every package reference or annotation as a violation unless the evidence shows that a framework concern shapes or is required by core behavior. General language or platform constructs are not framework concerns by themselves unless the reviewed material shows framework-specific coupling.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant framework concerns and the available evidence supports that framework-specific APIs and configuration remain outside application core behavior.

# Fail Conditions

Use `Fail` when direct evidence shows that application core behavior depends on framework-specific APIs, framework lifecycle mechanisms, framework configuration types, framework-specific annotations or attributes, or framework integration contracts within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, indirect, or ambiguous framework concern leakage into the core, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include application core behavior or framework concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the application core scope, identify framework concerns, inspect whether those concerns appear in core behavior, distinguish framework-specific coupling from general platform use, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when framework concerns shape central application behavior, broad core contracts, lifecycle assumptions, or stable architectural boundaries. Assign lower severity when the framework coupling is narrow, isolated, indirect, or limited to peripheral core behavior. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant core scope, framework concern, and coupling to core behavior. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.

# Dependencies

HEX-007. This is a conceptual dependency for shared core-adapter dependency vocabulary, not a procedural prerequisite for evaluating this rule.
