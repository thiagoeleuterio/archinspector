# Rule ID

HEX-006

# Title

Ports must be owned by the application core

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether boundary abstractions are defined from the application core's needs rather than from adapter details within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes application core behavior and one or more inbound or outbound ports, boundary abstractions, adapter-facing contracts, or equivalent interfaces used to connect the core with external actors or external systems. The rule is relevant only when there is enough context to distinguish core needs from adapter-specific details.

# Rule Statement

Ports must be owned by the application core.

# Rationale

When ports are shaped or owned by adapters, the boundary can reflect delivery mechanisms, infrastructure APIs, persistence models, transport protocols, or external integration details instead of the needs of the application core. This weakens core autonomy and makes adapter changes more likely to alter application behavior or contracts.

# Evidence Required

Evaluation requires traceable evidence that identifies the application core scope, the reviewed ports or equivalent boundary abstractions, and the adapters or external mechanisms related to those ports. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, or reviewed code excerpts showing where ports are declared, which components depend on them, and whether their members express core needs or adapter details. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between application core responsibilities, ports, and adapters. Then evaluate whether the port contracts are declared within the core boundary or otherwise controlled by the core's application-facing needs, and whether their shape is independent from concrete adapter mechanisms. Interpret evidence conservatively, and do not infer ownership from naming, intended architecture, or directory labels alone. A port should be considered core-owned when its dependency direction, placement, and contract shape support the conclusion that adapters conform to the core rather than the core conforming to adapters.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant ports and the available evidence supports that boundary abstractions are owned by the application core and shaped by core needs rather than adapter details.

# Fail Conditions

Use `Fail` when direct evidence shows that ports are declared in, controlled by, or shaped primarily by adapters, infrastructure mechanisms, delivery mechanisms, persistence contracts, transport contracts, external APIs, or other adapter-specific details within the reviewed scope.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, implicit, or ambiguous port ownership, or when a port mostly expresses core needs but includes some adapter-shaped details, and the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include application core behavior, ports, equivalent boundary abstractions, adapters, external actors, or external system interactions relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify the application core scope, identify ports or equivalent boundary abstractions, inspect where those abstractions are owned, determine whether their shape reflects core needs or adapter details, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when adapter-owned ports affect central application behavior, broad inbound or outbound surfaces, or stable architectural boundaries. Assign lower severity when the issue is narrow, isolated, or limited to peripheral boundary contracts. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant core scope, port ownership, dependency relationships, and contract shape. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.
