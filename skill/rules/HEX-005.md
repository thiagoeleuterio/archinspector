# Rule ID

HEX-005

# Title

Outbound adapters must implement outbound ports

# Category

Hexagonal Architecture

# Status

Active

# Intent

Evaluate whether outbound adapters provide external system integrations by implementing outbound ports outside the application core within the reviewed scope.

# Applicability

Apply this rule when the reviewed material includes outbound ports and outbound adapters, infrastructure integrations, or external-system-facing components that satisfy application core outbound needs. The rule is relevant only when there is enough context to distinguish outbound ports from their adapter implementations.

# Rule Statement

Outbound adapters must implement outbound ports.

# Rationale

Keeping outbound adapter implementations behind outbound ports prevents external integration details from becoming part of the core boundary, preserves adapter substitutability, and keeps infrastructure changes localized outside the application core.

# Evidence Required

Evaluation requires traceable evidence that identifies the reviewed outbound ports and the outbound adapters or external-system-facing components associated with them. Direct evidence may include project or module references, dependency declarations, imports, namespace relationships, type dependencies, constructor dependencies, method behavior, or reviewed code excerpts showing whether outbound adapters satisfy outbound ports while remaining outside the application core. Naming conventions may support interpretation but must not be treated as conclusive evidence without corroborating structural or behavioral evidence.

# Evaluation Guidance

Determine first whether the reviewed material supports a defensible distinction between outbound ports and outbound adapters. Then evaluate whether concrete external integration behavior is provided by adapters that implement or satisfy the outbound ports rather than by the core itself or by components that bypass the defined boundary. Interpret evidence conservatively, and do not infer implementation relationships from naming, intended architecture, or unavailable material.

# Pass Conditions

Use `Pass` when the reviewed scope is sufficient to evaluate relevant outbound ports and adapters and the available evidence supports that outbound adapters implement or satisfy outbound ports outside the application core.

# Fail Conditions

Use `Fail` when direct evidence shows that outbound adapters bypass outbound ports, fail to implement or satisfy the relevant outbound ports, or place concrete external integration behavior inside the application core.

# Warning Conditions

Use `Warning` when evidence indicates partial, inconsistent, implicit, or ambiguous implementation of outbound ports by outbound adapters, but the available material does not justify a definitive `Fail`.

# Not Applicable Conditions

Use `Not Applicable` when the reviewed scope does not include outbound ports, outbound adapters, external-system-facing components, or outbound integration concerns relevant to this rule.

# Not Enough Evidence Conditions

Use `Not Enough Evidence` when the rule may be relevant but the available material does not provide enough evidence to identify outbound ports, identify outbound adapters, inspect implementation relationships, determine whether adapters remain outside the core, or resolve conflicting signals.

# Severity Guidance

Assign higher severity when bypassed or missing outbound adapter implementations affect central integration flows, broad external-system surfaces, or stable architectural boundaries. Assign lower severity when the issue is narrow, isolated, or limited to peripheral integration paths. Severity must reflect the demonstrated architectural impact within the reviewed scope.

# Confidence Guidance

Use `Confirmed` when direct evidence clearly establishes the relevant ports, adapters, implementation relationships, and core boundary. Use `Likely` when multiple evidence points support the evaluation but direct confirmation is incomplete. Use `Possible` when evidence is limited, indirect, or partially ambiguous. Use `Not Enough Evidence` when the material cannot support a reliable conclusion. Naming alone must not produce `Confirmed` confidence.
